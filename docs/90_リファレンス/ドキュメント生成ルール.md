# 📋 ドキュメント作成・管理ルール

## 🌟 目的

本ドキュメントは、プロジェクトにおける設計ドキュメントの整備・追跡を統一的に行うためのルールを定めます。要件 → 基本設計 → 詳細設計 → 実装の流れを明示的に続りつけ、トレーサビリティを確保します。

---

## 🗂️ ディレクトリ構成と分類

```
docs/
├―― 01_企画構想/
├―― 02_要件定義/       # FR-XXX：機能要件 / NFR-XXX：非機能要件
├―― 03_基本設計/       # API設計、構成図、DB設計（要件IDで命名）
├―― 04_詳細設計/       # 各設計詳細（画面、バリデーションなど）
├―― 05_実装/
├―― 06_テスト/
├―― 07_運用設計/
└―― 99_その他/
```

---

## 🏋️ 命名規則

- 各ドキュメントは「要件ID」を先頭につけて命名します
- 複数の要件にまたがる場合は `_FR-001_FR-002_...` のように列挙

例：

```
02_要件定義/
  FR-001_ユーザー登録機能.md

03_基本設計/
  FR-001_API設計.md
  FR-001_DB設計.md

04_詳細設計/
  FR-001_画面仕様.md
  FR-001_バリデーション設計.md
```

---

## 🔗 トレーサビリティマトリクス

以下のように要件 → 設計 → 実装の繋がりを一覧管理します（Excel / Markdown など）

| 要件ID | 要件名       | 基本設計                  | 詳細設計                       | 実装ファイル           |
|----------|----------------|-----------------------------|----------------------------------|------------------------------|
| FR-001   | ユーザー登録 | `03/FR-001_API設計.md`  | `04/FR-001_画面仕様.md`         | `src/routes/signup.ts`       |
| FR-002   | ログイン       | `03/FR-002_API設計.md`  | `04/FR-002_処理ロジック.md` | `src/routes/login.ts`        |

---

## 💬 コメント/メタタグの埋め込み（オプション）

Markdownやコードに以下のように記述します

```markdown
<!-- @requirement: FR-001 -->
<!-- @basic_design: 03/FR-001_API設計.md -->
```

```ts
// @requirement: FR-001
// signup.ts - ユーザー登録用エンドポイント
```

---

## 🧰 拡張運用アイデア

- GitHub PRテンプレートに「関連要件ID」を含める
- Chatbotで「このコードは元になった要件は？」と聞ける構成
- CIで未対応要件の自動検知も可

---

## 📝 補証

- 非機能要件は `NFR-001` などのスタイルで調整
- 機能ごとにサブディレクトリ分類も有効

# IPA共通フレーム対応表

## 概要

aiDev フレームワークと IPA（情報処理推進機構）共通フレーム2013 の対応関係を示す。

**目的**:
- 業界標準との整合性確認
- 第三者への説明時の参照

**注意**: aiDev は IPA 共通フレームを参考としつつ、AI支援開発に最適化した独自フレームワークです。完全な準拠ではなく、参考対応です。

---

## 1. フェーズ対応表

### aiDev フェーズ ↔ IPA プロセス

| aiDev フェーズ | IPA 共通フレーム | 対応レベル | 備考 |
|--------------|-----------------|----------|------|
| 企画フェーズ | 企画プロセス | 部分対応 | ビジネス要件の明確化 |
| 要件定義フェーズ | 要件定義プロセス | 高対応 | 機能/非機能要件の定義 |
| 設計フェーズ | システム方式設計、ソフトウェア方式設計 | 高対応 | アプリ/インフラ分離 |
| 実装フェーズ | ソフトウェア詳細設計、構築 | 高対応 | コード実装 |
| テストフェーズ | ソフトウェア結合、システム結合、システム適格性確認 | 高対応 | テスト実施 |
| デプロイフェーズ | 導入・受入れ支援 | 部分対応 | IaC によるデプロイ |

### 対応レベルの定義

| レベル | 定義 |
|-------|------|
| 高対応 | IPA プロセスの主要なアクティビティをカバー |
| 部分対応 | IPA プロセスの一部をカバー（スコープが異なる） |
| 参考 | IPA プロセスを参考にしているが、独自アプローチ |

## 4. テストレベル対応

### aiDev テスト ↔ IPA テストレベル

| aiDev テストレベル | 担当 | IPA テストレベル | 備考 |
|------------------|------|-----------------|------|
| 単体テスト | Coder | ソフトウェアユニットテスト | 関数・クラス単位 |
| 結合テスト | QA | ソフトウェア結合テスト | モジュール間連携 |
| システムテスト | QA | システム結合テスト | システム全体 |
| 受け入れテスト | QA + PM | システム適格性確認テスト | ユーザー視点 |
| 性能テスト | QA + SRE | 性能テスト | 負荷テスト、ストレステスト |
| セキュリティテスト | QA + SRE | セキュリティテスト | 脆弱性診断 |

### テスト責務の分担

```
Coder:
  - 単体テスト（実装と同時に作成）
  - カバレッジ 80% 以上

QA:
  - 結合テスト
  - システムテスト
  - 受け入れテスト
  - テスト計画・設計

SRE:
  - インフラテスト（IaC の検証）
  - 性能テスト（インフラ観点）
  - 障害復旧テスト
```

## 6. aiDev 独自の概念

以下は aiDev 独自の概念で、IPA 共通フレームには直接対応するものがない：

| aiDev 概念 | 説明 | 位置づけ |
|-----------|------|---------|
| クロスレビュー | 作成者と異なるエージェントがレビュー | 品質向上施策 |
| インターフェース仕様書 | App⇔Infra の境界面定義 | 整合性担保 |
| 実装チェッカー | 設計書参照の強制 | 品質向上施策 |
| 二重実装防止 | 既存コード確認の強制 | 品質向上施策 |
| IaC コード | インフラのコード化 | 現代的開発手法 |
| 技術標準 | コーディング規約等 | 品質標準化 |

# aiDev 開発プロセスガイドライン

## 1. 基本方針

aiDevプロジェクトの開発においては、以下の原則を徹底します：

- **段階的開発**: 一気に実装せず、小さな単位で開発とテストを繰り返す
- **問題の局所化**: 問題の切り分けを容易にするため、変更範囲を明確に限定する
- **継続的なフィードバック**: 実装→確認→フィードバックのサイクルを短期間で回す
- **実装前の設計レビュー**: コードを書く前に設計の段階でレビューを行う

## 2. タスク分解の方法

### 2.1 最小実行可能単位への分解

大きな機能や構築物は、以下の手順で最小実行可能単位に分解します：

1. **概念実証(PoC)単位への分解**:
   - 例: IaCでの3環境構築 → まず1つの最小VPC構築のみを実装
   - 例: 認証機能の実装 → まず静的ユーザーでのログインのみを実装

2. **機能の積み上げ**:
   - 最小単位が動作することを確認してから機能を追加
   - 例: 最小VPC → ネットワークACL追加 → セキュリティグループ追加

3. **分解の具体例**:

#### インフラ構築の例:
```
大きな目標: 本番/ステージング/開発の3環境をCloudFormationで構築

分解ステップ:
1. 最小VPC(サブネットのみ)を1環境分だけ構築し動作確認
2. VPCに必要なセキュリティ設定を追加し動作確認
3. パラメータ化して複数環境に対応
4. 他のコンポーネント(EC2, RDS等)を一つずつ追加
```

#### アプリケーション開発の例:
```
大きな目標: Bedrockを使った対話インターフェースの実装

分解ステップ:
1. Bedrockへの単純なクエリ送信と応答表示のみを実装
2. コンテキスト管理機能を追加
3. ナレッジベース検索連携を追加
4. UIの改善と応答表示の最適化
```

## 3. レビュープロセス

### 3.1 デザインレビュー

- **タイミング**: 実装前の設計段階
- **目的**: 設計の妥当性、技術選定の適切さ、非機能要件への対応を確認
- **参加者**: アーキテクト、テックリード、必要に応じて開発メンバー
- **フォーマット**: 
  - 設計文書のレビュー（アーキテクチャ図、データフロー等）
  - 考慮した代替案と選定理由
  - リスクと対策

### 3.2 コードレビュー

- **タイミング**: 実装後、プルリクエスト時
- **目的**: コード品質、ベストプラクティス準拠、セキュリティ、テスト網羅性の確認
- **参加者**: 開発メンバー、テックリード
- **フォーマット**:
  - 変更箇所の明示
  - ユニットテストの結果
  - 変更による影響範囲

### 3.3 継続的なブレスト

- **頻度**: 週1回以上
- **目的**: 課題の早期発見、解決策の検討、知識共有
- **参加者**: 開発チーム全体
- **フォーマット**:
  - 現在の課題と進捗の共有（15分）
  - 特定課題に関するブレスト（30分）
  - アクションアイテムの決定（15分）

## 4. IaC開発のベストプラクティス

### 4.1 CloudFormation開発手順

1. **単一リソースからの開始**:
   - まず単一リソース（VPCやS3バケット等）のテンプレートを作成して動作確認
   - 検証方法: `aws cloudformation validate-template`で構文チェック
   - 検証方法: `aws cloudformation deploy`で実際にデプロイして確認

2. **段階的な拡張**:
   - 動作確認済みのテンプレートに少しずつリソースを追加
   - 各段階でテスト: 追加→検証→コミットのサイクルを徹底

3. **パラメータ化と再利用**:
   - 基本構造が安定した後にパラメータ化
   - 環境ごとのパラメータファイルを用意
   - 共通部分はNestedスタックとして切り出し

4. **変更管理**:
   - ChangeSetを活用して変更影響を事前確認
   - 問題があれば修正してChangeSetを再作成

### 4.2 IaC開発時の注意点

- **依存関係の明示**: `DependsOn`属性を適切に使用
- **過度な結合を避ける**: リソース間の不必要な依存を作らない
- **シークレット管理**: パスワードなどの機密情報はSecretsManagerを使用
- **ドリフト検出**: 定期的なドリフト（手動変更）検出を実施
- **ロールバック考慮**: 失敗時のロールバック方法を事前に計画

## 5. ソフトウェア開発のベストプラクティス

### 5.1 フロントエンド開発

- **コンポーネント分割**: 再利用可能な小さなコンポーネントに分割
- **状態管理**: グローバル状態と局所状態を適切に分離
- **プログレッシブ拡張**: 基本機能から段階的に高度な機能を追加
- **モック活用**: バックエンドが未実装の場合はモックを活用

### 5.2 バックエンド開発

- **単一責任**: 各関数・モジュールは単一の責任を持つように設計
- **ユニットテスト**: ビジネスロジックには必ずユニットテストを作成
- **環境分離**: 開発/テスト/本番環境の分離と設定の外部化
- **ログ管理**: 適切なログレベルと構造化ログの実装

## 6. 実行・検証時のチェックリスト

### 6.1 実行前チェック

- [ ] ローカルでの構文チェックを実施
- [ ] 依存コンポーネントが存在するか確認
- [ ] 必要な権限・認証情報が設定されているか確認
- [ ] 実行計画（何が変更されるか）を確認

### 6.2 実行後チェック

- [ ] 意図した通りに実行されたか検証
- [ ] ログに異常がないか確認
- [ ] 関連リソース/コンポーネントが正常に動作するか確認
- [ ] パフォーマンスは要件を満たしているか確認

## 7. トラブルシューティング

### 7.1 問題特定手順

1. **症状の正確な把握**: エラーメッセージ、発生条件を明確にする
2. **範囲の特定**: 直近の変更点から調査を開始
3. **単純化**: 問題を再現する最小のケースを作成
4. **ログの分析**: 関連するログを時系列で分析

### 7.2 共通問題と解決方法

- **IaCデプロイ失敗**: CloudFormationのイベントログで詳細を確認
- **API接続エラー**: ネットワーク設定とIAM権限を確認
- **パフォーマンス問題**: リソース使用率とボトルネックを特定
- **データ整合性問題**: トランザクション境界とエラーハンドリングを確認

## 8. ドキュメント化

開発プロセス中は以下のドキュメントを継続的に更新します：

- **設計ドキュメント**: 実装中に明らかになった詳細を反映
- **トラブルシューティングガイド**: 発生した問題と解決策を記録
- **運用手順**: 実際の運用に必要な手順を段階的に整備
- **レッスンズラーンド**: プロジェクト中に学んだ教訓を記録

## 9. プロセス改善

- **レトロスペクティブ**: スプリント終了ごとにプロセス改善点を議論
- **改善の実装**: 合意した改善点を次のスプリントから適用
- **メトリクス測定**: 改善の効果を定量的に測定（例：バグ発生率の変化）

## 10. バージョン管理とコミットルール

### 10.1 コミットメッセージの形式

```
[タイプ] 概要（50文字以内）

詳細な説明（必要な場合）

関連チケット: #123
```

### 10.2 タイプ分類

- `[docs]` - ドキュメント関連の変更
- `[feat]` - 新機能の追加
- `[fix]` - バグ修正
- `[refactor]` - リファクタリング（機能変更なし）
- `[infra]` - インフラストラクチャコードの変更
- `[test]` - テスト関連の修正/追加
- `[chore]` - ビルドプロセス、ツール関連の変更

### 10.3 ブランチ戦略

- `main` - 安定版、リリース用
- `develop` - 開発中の統合ブランチ
- `feature/XX` - 新機能開発用
- `fix/XX` - バグ修正用
- `docs/XX` - ドキュメント更新用

### 10.4 コミットの粒度

- 論理的に完結した1つの変更を1コミットとする
- 大きな変更は複数の小さなコミットに分割する
- 「動作する状態」でコミットする

### 10.5 プルリクエスト

- 機能単位でプルリクエストを作成
- レビュー担当者を明示
- CI/CDパイプラインのテスト結果を確認

# 02-1 aiDev ナレッジベース詳細設計

## 1. 概要

aiDevシステムにおけるナレッジベースは、AIエージェントの専門知識ベースであるとともに、ユーザーとの対話の履歴や生成コンテンツを保存・活用するプラットフォームの基盤となるものです。本ドキュメントでは、ナレッジベースの構造、データモデル、そして自己進化型システムとしての実装詳細を定義します。

## 2. ナレッジベースのアーキテクチャ

### 2.1 全体アーキテクチャ

ナレッジベースは以下の層から構成されます：

1. **静的ナレッジ層**：予め用意された専門知識
   - AWS設計・構築ベストプラクティス (MCP経由でアクセス)
   - 業界別アーキテクチャパターン
   - 技術リファレンス

2. **動的ナレッジ層**：対話を通じて蓄積される知識
   - 顧客固有コンテキスト
   - 対話から抽出された要件・課題
   - 過去の設計事例

3. **メタナレッジ層**：知識の管理と最適化のための情報
   - 参照頻度・有用性の統計
   - ナレッジ間の関連性
   - バージョン管理情報

### 2.2 ハイブリッドナレッジアクセスアーキテクチャ

ナレッジベースは、MCPとS3のハイブリッドアーキテクチャで実装します：

#### 2.2.1 MCPベースのAWS関連ナレッジ

AWS関連の専門知識はMCP (Model Customization Provider) を通じて提供されます：

- **AWSサービス情報**: サービスの詳細、制限、APIリファレンス
- **ベストプラクティス**: AWS Well-Architected Framework、セキュリティベストプラクティス
- **アーキテクチャパターン**: 一般的なAWSアーキテクチャパターン、リファレンスアーキテクチャ
- **コスト最適化**: 料金情報、最適化戦略、節約プラン

企業固有の知識および生成コンテンツはS3に保存されます：

#### 2.2.3 ハイブリッドアクセスモデル

ナレッジアクセスの統合は以下のアプローチで実装します：

1. **優先順位付けアクセス**:
   - 企業固有情報 > AWS関連知識（MCP）> 一般的な技術知識
   - 企業固有コンテキストに基づいた知識検索の優先度調整

2. **統合検索インターフェース**:
   - 単一のAPIから両方のデータソースへのアクセスを提供
   - 複数ソースからの結果の統合と重複排除

3. **キャッシュ最適化**:
   - 頻繁にアクセスされるAWS情報のローカルキャッシュ
   - キャッシュの有効期限管理と定期的な更新

### 2.3 企業固有知識の高解像度対応

企業固有知識の高解像度対応とは、各企業・組織のコンテキスト、要件、制約などを詳細に構造化して保存し、AIエージェントが高精度で参照できるようにする仕組みです。

#### 2.3.1 高解像度データの構成要素

1. **ビジネスコンテキスト**
   - 業界位置づけ（競合・市場環境）
   - 重要業績評価指標（KPI）
   - 経営戦略・中期計画
   - 組織構造と意思決定プロセス

2. **IT戦略とガバナンス**
   - IT成熟度
   - 技術選定基準
   - セキュリティ方針
   - コンプライアンス要件（業界規制・社内規定）
   - 予算サイクルと承認プロセス

3. **テクニカルコンテキスト**
   - 現行システム構成
   - 技術的負債
   - 標準技術スタック
   - インフラ制約
   - 開発/運用チームのケイパビリティ

4. **プロジェクト/要件パターン**
   - 過去プロジェクトの成功/失敗要因
   - 典型的な要件パターン
   - 意思決定の傾向
   - 優先順位付けの傾向

#### 2.3.2 高解像度データの取得・更新メカニズム

1. **初期データ収集**
   - 初期セッションでの明示的質問
   - 提供文書からの抽出
   - 対話からの暗黙的情報抽出

2. **自動更新メカニズム**
   - 対話中の矛盾検出と解決
   - 新規情報の自動分類と統合
   - 定期的な情報鮮度チェック
   - 確信度スコアリングと検証

3. **差分管理**
   - 情報の時系列変化追跡
   - 変更理由の推定・記録
   - 有効期限管理

## 3. 自己進化型ナレッジベースの実装

### 3.1 ナレッジ抽出・構造化プロセス

対話からナレッジを抽出し、構造化するプロセスは以下の通りです：

1. **対話内容の分析**
   - 会話から重要情報を特定
   - メタデータ（確信度、コンテキスト）の付与
   - 既存ナレッジとの整合性チェック
   - ユーザーによる承認

2. **情報の構造化**
   - スキーマに基づくJSON形式への変換
   - 階層的カテゴリへの分類
   - 参照リンクの生成

3. **ナレッジグラフへの統合**
   - 新規情報と既存情報の関連付け
   - 矛盾解決と優先度決定
   - メタデータの更新

4. **ハイブリッドソース参照**
   - S3ストレージからの企業固有情報の参照
   - MCP経由でのAWS関連知識の取得
   - 複数ソースからの知識の統合と優先順位付け

#### 3.1.1 抽出アルゴリズム概要

```
1. 対話テキストから企業固有情報の候補を特定
   - 業界用語、システム名、制約条件などのキーワード検出
   - 文脈から重要度を評価

2. 候補情報を既存カテゴリに分類
   - カテゴリ別分類器を使用
   - 確信度スコア付与

3. 構造化データに変換
   - カテゴリ別テンプレートに基づく変換
   - 関連メタデータの付与

4. 既存データとの整合性確認
   - 既存データとの重複・矛盾チェック
   - 時間的前後関係の考慮
   - 確信度に基づく採用判断
```

#### 3.1.2 MCP参照フロー

AWS関連の情報が必要な場合、以下のフローでMCPから知識を取得します：

```
1. キーワード・意図分析
   - 会話からAWS関連キーワードの抽出
   - AWS関連の質問・要件の分類（サービス情報/ベストプラクティス/アーキテクチャ）

2. MCP APIリクエスト構築
   - 適切なクエリパラメータの生成
   - 企業コンテキストに基づく関連性フィルタの適用

3. レスポンス処理
   - MCPからのレスポンスの構造化
   - 関連性スコアリングと優先順位付け
   - ローカルキャッシュへの保存（頻繁に利用される情報）

4. S3ナレッジとの統合
   - 企業固有コンテキストに基づくフィルタリング
   - シナリオに関連する情報の優先順位付け
   - 統合された回答の生成
```

このMCP参照フローは、企業固有の要件やコンテキストに基づいて、一般的なAWS知識を特定の顧客シナリオに適用する際に特に重要です。また、MCP経由でアクセスすることで、常に最新のAWSサービス情報やベストプラクティスを参照できます。

### 3.2 ナレッジ最適化メカニズム

ナレッジベースの効率と精度を継続的に向上させるメカニズムとして、以下を実装します：

1. **使用頻度・有効性の追跡**
   - ナレッジ項目ごとの参照頻度記録
   - エージェント応答への貢献度評価
   - ユーザーフィードバックとの相関分析

2. **自動整理・最適化**
   - 低価値情報の降格・アーカイブ
   - 高価値情報の昇格・強調
   - 関連情報のクラスタリング強化

3. **鮮度管理**
   - 時間経過に伴う確信度減衰
   - 更新トリガーの設定
   - 検証必要フラグの管理

#### 3.2.1 最適化パイプライン

```
Daily:
  - 新規追加ナレッジの初期分類と関係付け
  - 使用統計の更新

Weekly:
  - キーワード検索効率の評価と最適化
  - クラスター分析と関連性強化

Monthly:
  - 低使用ナレッジの評価とアーカイブ候補選定
  - 鮮度指標に基づく更新候補の特定
  - パフォーマンスメトリクスレポート生成
```

### 3.3 バックエンド処理フロー

ナレッジベースの更新・最適化を支えるバックエンド処理フローを以下に示します：

1. **対話後処理フロー**
   ```
   対話完了 → 
   Lambda:SessionAnalyzer →  
   SQS:KnowledgeExtractionQueue → 
   Lambda:KnowledgeExtractor → 
   S3:dynamic/organizations/{org_id}/...
   ```

2. **定期最適化フロー**
   ```
   EventBridge:ScheduledTrigger → 
   Lambda:KnowledgeOptimizer → 
   S3:変更適用 → 
   SNS:OptimizerNotification
   ```

3. **検索インデックス更新フロー**
   ```
   S3:PUT/DELETE イベント → 
   Lambda:IndexUpdater → 
   OpenSearch:インデックス更新
   ```

4. **ハイブリッドナレッジ検索フロー**
   ```
   対話クエリ → 
   Lambda:HybridKnowledgeRouter → 
   [条件分岐: AWS関連?]
     Yes → MCP APIリクエスト → MCP結果処理 → 
     No → S3/OpenSearch検索 →
   結果統合 → 
   クライアントレスポンス
   ```

5. **MCPキャッシュ更新フロー**
   ```
   EventBridge:DailyTrigger → 
   Lambda:MCPCacheUpdater → 
   [頻繁に使用されるAWS知識の特定] →
   MCP APIリクエスト →
   DynamoDB:MCPCache保存
   ```

## 5. 評価指標とモニタリング

ナレッジベースの効果と品質を評価するための指標を以下に定義します：

### 5.1 品質指標

- **カバレッジスコア**: 顧客質問に対するナレッジベースのカバー率
- **正確性スコア**: 提供情報の正確さの評価
- **鮮度指標**: コンテンツの最新性評価
- **構造化レベル**: 情報の構造化度合い

### 5.2 効率指標

- **検索レイテンシ**: 情報検索にかかる平均時間
- **抽出効率**: 対話からの情報抽出成功率
- **ストレージ効率**: 重複排除後の実効ストレージ率

### 5.3 ビジネス指標

- **エージェント性能向上率**: ナレッジベース強化によるエージェント回答品質向上
- **コンテキスト理解スコア**: 企業固有コンテキストの理解度
- **再利用率**: 過去のナレッジが新規対話で再利用される率

## 8. エージェントプロセス管理

aiDevシステムにおけるエージェントのタスクプロセス（開発プロセス、コーディングプロセス、要件定義プロセスなど）は、ナレッジベースとは別に、中央管理型プロセス定義アプローチで管理します。

### 8.1 プロセス定義の構造

```
プロセス管理サービス
├── コアプロセス定義（共通）
│   ├── 開発プロセス_v1.json
│   ├── 開発プロセス_v2.json（更新版）
│   ├── 要件定義プロセス_v1.json
│   └── コーディングプロセス_v1.json
│
└── テナント固有カスタマイズ
    ├── tenant_001/
    │   └── 開発プロセス_override.json（差分のみ）
    └── tenant_002/
        └── 要件定義プロセス_override.json（差分のみ）
```

### 8.2 プロセス優先順位

1. **優先順位モデル**:
   - ユーザー固有プロセス定義 > コアプロセス定義
   - ユーザー固有プロセスに定義がない場合、コアプロセスを適用
   - 競合する場合はユーザー固有プロセスが優先

2. **継承モデル**:
   - コアプロセスはフレームワークやガイドラインとして機能
   - ユーザー固有プロセスはコアプロセスを拡張・カスタマイズ
   - オーバーライドは最小限に留め、差分のみを管理

### 8.3 プロセス更新メカニズム

1. **コアプロセスの更新**:
   - 一元管理されたソースからの更新
   - バージョン管理による互換性確保
   - CI/CDパイプラインを通じた展開

2. **ユーザー固有プロセスの更新**:
   - テナントごとの個別更新
   - コアプロセス更新時の影響評価
   - 差分管理による効率化

3. **更新フロー**:
   ```
   コアプロセス更新 → 互換性チェック → 全テナントへの展開 → 
   テナント固有設定との動的マージ → 適用
   ```

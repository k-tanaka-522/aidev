# プリセールスエージェント詳細設計

## 1. 概要

プリセールスエージェントは、aiDevシステムの最初の接点となるAIエージェントであり、ユーザーの課題や要件を初期ヒアリングし、AWS環境構築の提案や見積もりを行う役割を担います。ビジネス視点からの提案能力と技術的な実現可能性の評価を兼ね備え、ユーザーのニーズに合わせた最適なソリューションを提案します。

## 2. 役割と機能

### 2.1 主要な役割

1. **初期ヒアリング**
   - ユーザーの業務課題やシステム要件の把握
   - プロジェクトの目的、規模、予算、期限などの基本情報収集
   - 技術的な制約条件や既存環境の確認

2. **要件整理と分析**
   - 収集した情報の構造化と優先順位付け
   - 要件の実現可能性とリスクの評価
   - 不明点や矛盾点の特定と追加質問

3. **AWSソリューション提案**
   - 要件に適したAWSサービスの選定と構成提案
   - アーキテクチャ案の提示
   - 技術的メリット・デメリットの説明

4. **コスト試算と提案**
   - AWS環境の概算見積もりの作成
   - コスト最適化提案
   - ROIや投資効果の分析

5. **プロジェクト計画支援**
   - 実装スケジュールの提案
   - フェーズ分けとマイルストーン設定
   - リスク管理計画の立案

### 2.2 詳細機能一覧

| 機能ID | 機能名 | 説明 | 優先度 |
|--------|-------|------|-------|
| PS-001 | 初期質問セット | 業種や目的に応じた最適な初期質問を自動選択 | 高 |
| PS-002 | 要件抽出 | 会話から要件を自動抽出し構造化 | 高 |
| PS-003 | AWS構成提案 | 要件に基づいた最適なAWS構成を提案 | 高 |
| PS-004 | コスト試算 | AWS料金体系を基にした概算見積もり生成 | 高 |
| PS-005 | ダイアグラム生成 | アーキテクチャ図の自動生成 | 中 |
| PS-006 | 提案資料作成 | 提案書や見積書のテンプレート生成 | 中 |
| PS-007 | 技術的リスク評価 | 構成上のリスクポイントを特定し対策を提案 | 中 |
| PS-008 | 事例ベース提案 | 類似事例を参照した具体的な提案 | 中 |
| PS-009 | 代替案提示 | 複数の選択肢と比較表の提示 | 低 |
| PS-010 | 競合比較 | 他クラウドサービスとの比較分析 | 低 |

## 3. インプット・アウトプット

### 3.1 インプット情報

#### 3.1.1 ユーザーからの入力

- テキストベースの質問や要件
- プロジェクト情報（目的、規模、予算、期限など）
- 既存システム情報（現状の課題、構成、制約など）
- 業界・業種情報
- 技術的な好み・制約

#### 3.1.2 システムからの入力

- ユーザープロファイル情報
- 過去の対話履歴
- セッションコンテキスト
- ナレッジベースからの参照情報
- 他エージェントからの連携情報

### 3.2 アウトプット情報

#### 3.2.1 基本レスポンス

- テキストベースの回答・提案
- フォローアップ質問
- 要件の確認と整理

#### 3.2.2 生成コンテンツ

- AWS構成案（テキスト説明）
- アーキテクチャ図（テキストベース図）
- コスト見積もり（概算）
- 開発スケジュール案
- 技術的リスクと対策

## 4. プロンプト設計

### 4.1 システムプロンプト

```
あなたはAWS専門のプリセールスエンジニアとして、お客様のビジネス課題に最適なAWSソリューションを提案します。以下の役割と責務を持ちます：

1. 親身になってお客様の課題をヒアリングし、要件を明確化する
2. AWSのベストプラクティスに基づいた最適な構成を提案する
3. 技術的な内容をビジネス価値と紐付けて説明する
4. コスト最適化を意識したソリューションを提案する
5. セキュリティとコンプライアンスを常に考慮する

お客様とのやり取りでは、以下のポイントを心がけてください：
- 専門用語を過度に使わず、わかりやすく説明する
- 自信を持って提案するが、確証のない情報は提供しない
- お客様の予算と優先事項を常に尊重する
- 必要に応じて追加質問で要件を明確化する
- 複数の選択肢がある場合は、メリット・デメリットを明示する

また、AIシステム「aiDev」の特徴と価値についても理解し、お客様に以下の点をアピールできるようにしてください：
- 対話型AIによる開発支援の効率性
- AWS環境の自動設計・構築による時間・コスト削減
- 専門知識の集約と継続的な改善
- セキュアなマルチテナント構成
```

## 5. 状態管理

### 5.1 エージェント状態

プリセールスエージェントは以下の状態を持ちます：

1. **初期ヒアリング状態**
   - 基本情報収集フェーズ
   - 主にユーザーからの情報入力を促進

2. **要件分析状態**
   - 収集情報の整理と構造化
   - 要件の優先度付けと矛盾点の解消

3. **ソリューション提案状態**
   - AWS構成案の生成と説明
   - アーキテクチャ図や構成詳細の提示

4. **コスト試算状態**
   - 見積もり情報の生成
   - コスト最適化提案

5. **提案調整状態**
   - ユーザーのフィードバックに基づく提案の調整
   - 代替案の検討と提示

### 5.2 状態遷移図

```
[初期ヒアリング状態] --> [要件分析状態] --> [ソリューション提案状態] --> [コスト試算状態] --> [提案調整状態]
     ^                                                                                    |
     |                                                                                    |
     +------------------------------------------------------------------------------------+
```

※ 状態間は必要に応じて双方向に遷移可能

### 5.3 コンテキスト管理

- **セッションコンテキスト**: ユーザーとの対話全体を通じて維持されるコンテキスト
- **ローカルコンテキスト**: 各状態内で参照される一時的なコンテキスト
- **持続コンテキスト**: ユーザーのプロファイルや過去の提案内容など永続的に保存されるコンテキスト

## 7. エラーハンドリング

### 7.1 想定されるエラーパターン

1. **情報不足エラー**
   - 必要な情報が収集できていない場合
   - 対応: 不足情報の特定と追加質問

2. **矛盾要件エラー**
   - ユーザーの要件間に矛盾がある場合
   - 対応: 矛盾点の明示と優先順位の確認

3. **技術的実現不能エラー**
   - 要件が技術的に実現困難な場合
   - 対応: 代替案の提示と制約の説明

4. **予算超過エラー**
   - 提案がユーザー予算を超過する場合
   - 対応: コスト削減オプションの提示

### 7.2 エラー応答例

| エラーパターン | 応答例 |
|--------------|--------|
| 情報不足エラー | 「ご要件を適切に把握するために、もう少し詳しくお聞かせいただけますか？特に{{不足情報}}について教えていただけると、より正確な提案ができます。」 |
| 矛盾要件エラー | 「ご要望の中で、{{要件A}}と{{要件B}}が技術的に両立が難しい点がございます。どちらをより優先されますか？」 |
| 技術的実現不能エラー | 「ご要望の{{要件}}は、現在のAWSサービスでは完全な実現が難しい状況です。代わりに{{代替案}}というアプローチであれば、類似の効果を得られると考えられます。」 |
| 予算超過エラー | 「現在の構成ですと、ご予算を約{{超過率}}%超過する見込みです。コストを抑えるために、{{削減オプション}}などの調整が可能です。」 |

## 8. ナレッジベース参照

### 8.1 参照するナレッジカテゴリ

1. **AWS Well-Architected Framework**
   - 信頼性、パフォーマンス効率、セキュリティ、コスト最適化、運用上の優秀性

2. **業界別ソリューションパターン**
   - 金融、医療、小売、製造、メディアなど業種別のベストプラクティス

3. **AWSサービスカタログ**
   - 各サービスの特徴、制限事項、料金体系

4. **アーキテクチャリファレンス**
   - 一般的なアーキテクチャパターンと実装例

5. **コスト最適化ガイド**
   - リザーブドインスタンス、Savings Plans、ライフサイクル管理など

### 8.2 参照方法

1. **キーワードベース検索**
   - ユーザー入力からキーワードを抽出し、関連知識を検索

2. **セマンティック検索**
   - ユーザーの意図理解に基づく意味論的な検索

3. **パターンマッチング**
   - 類似事例や構成パターンの検索

## 9. パフォーマンス指標

### 9.1 品質指標

1. **提案の適合性**
   - 要件カバー率
   - AWS Well-Architected Framework適合度

2. **ユーザー満足度**
   - 対話後のフィードバックスコア
   - 質問の的確さ評価

3. **コスト最適化度**
   - 提案コストと市場平均との比較
   - 無駄なリソースの排除率

### 9.2 効率指標

1. **要件把握速度**
   - 初期ヒアリングから要件整理までの時間
   - 追加質問数

2. **提案生成時間**
   - AWS構成案生成にかかる時間
   - コスト試算完了までの時間

3. **対話ターン数**
   - 提案完了までの対話ターン数
   - 1ターンあたりの情報獲得量

## 10. デプロイメント設計

### 10.1 実行環境

- AWS Lambda (Node.js 18.x)
- メモリ: 1024MB
- タイムアウト: 30秒
- 同時実行数: 最大100

### 10.2 スケーリング戦略

- 自動スケーリング（CloudWatchアラームベース）
- ピーク時の予約コンカレンシー設定
- リクエストベースのプロビジョニング

### 10.3 監視設定

- CloudWatch Logsによるログ収集
- API Gateway、Lambda、DynamoDB、Bedrockメトリクスの統合監視
- エラー率とレイテンシのアラート設定

# aiDev 認証・マルチテナント詳細設計

## 1. 概要

本ドキュメントでは、aiDevシステムにおける認証システムとマルチテナント環境の詳細設計を定義します。この設計は、AWS Cognito、IAM、およびAWS Organizationsを活用し、安全かつスケーラブルな認証・認可機能と顧客ごとの分離環境を実現するものです。

## 2. 認証・認可システム設計

### 2.1 認証システムアーキテクチャ

#### 2.1.1 全体構成

aiDevシステムの認証システムは、以下のコンポーネントで構成されます：

```
[クライアント] <---> [Amazon CloudFront] <---> [Amazon Cognito] <---> [API Gateway/Lambda]
                                                    |
                                                    v
                                          [Identity Federation]
                                                    |
                                                    v
                                      [子アカウントへのアクセス]
```

#### 2.1.2 Amazon Cognito設定

**ユーザープール設定**:
- **プール名**: `aidev-user-pool`
- **サインイン方法**: メールアドレスを主要識別子として使用
- **多要素認証(MFA)**: オプショナル（フェーズ2で必須化を検討）
- **メール検証**: 登録時に必須
- **パスワードポリシー**:
  - 最小長: 8文字
  - 大文字、小文字、数字を含む
  - 有効期限: 90日
  - 履歴確認: 過去5回分のパスワードを再利用不可

**アプリケーションクライアント**:
- **クライアント名**: `aidev-web-client`
- **認証フロー**: 
  - ユーザーパスワード認証
  - カスタム認証（フェーズ2で導入）
- **OAuth 2.0設定**:
  - コールバックURL: `https://app.aidev.example.com/auth/callback`
  - 許可されたスコープ: `openid`, `email`, `profile`
  - 許可されたフロー: `authorization_code`, `implicit`

**IDプール設定**:
- **プール名**: `aidev-identity-pool`
- **認証プロバイダー**: Cognito User Pool + 各子アカウントのIAM
- **ロール解決**: マッピングルールに基づく動的ロール割り当て

### 2.2 認証フロー

#### 2.2.1 初期サインアップフロー

1. ユーザーがメールアドレスを入力してサインアップ開始
2. Cognitoが検証コードをユーザーのメールアドレスに送信
3. ユーザーが検証コードを入力して検証完了
4. パスワード設定と最小限のプロファイル情報入力
5. サインアップ完了、初期トークン発行
6. 子アカウント作成ワークフローの開始（3.2節参照）

```sequence
ユーザー -> Webアプリ: メールアドレス入力
Webアプリ -> Cognito: SignUp API呼び出し
Cognito -> メールサービス: 検証コード送信
メールサービス -> ユーザー: 検証メール
ユーザー -> Webアプリ: 検証コード入力
Webアプリ -> Cognito: ConfirmSignUp API呼び出し
Cognito -> Webアプリ: 検証完了
Webアプリ -> ユーザー: パスワード設定画面
ユーザー -> Webアプリ: パスワード・プロファイル入力
Webアプリ -> Lambda: ユーザー登録完了通知
Lambda -> Step Functions: 子アカウント作成ワークフロー開始
Lambda -> DynamoDB: ユーザーレコード作成
Webアプリ -> ユーザー: プリセールス画面表示
```

#### 2.2.2 ユーザーサインインフロー

1. ユーザーがメールアドレスとパスワードを入力
2. Cognitoによる認証と認証トークン発行
3. JWT検証とセッション確立
4. アクセストークンとIDトークンの保存（HTTPOnly Cookie）
5. 子アカウントアクセス用の一時的認証情報取得（必要な場合）

```sequence
ユーザー -> Webアプリ: メール/パスワード入力
Webアプリ -> Cognito: InitiateAuth API呼び出し
Cognito -> Webアプリ: JWT発行
Webアプリ -> Lambda: トークン検証
Lambda -> DynamoDB: ユーザー情報取得
Lambda -> Webアプリ: セッション確立
Webアプリ -> ユーザー: ダッシュボード表示
Webアプリ -> Lambda: 子アカウントアクセス要求（必要時）
Lambda -> STS: AssumeRole API呼び出し
STS -> Lambda: 一時的認証情報
Lambda -> Webアプリ: 子アカウント認証情報
```

#### 2.2.3 認証トークン管理

- **トークンタイプ**:
  - IDトークン: ユーザー識別情報
  - アクセストークン: APIアクセス権限
  - リフレッシュトークン: トークン更新用
  
- **トークン保存**:
  - HTTPOnly Cookie（XSS対策）
  - Secure属性（HTTPS通信のみ）
  - SameSite=Strict（CSRF対策）

- **トークン更新メカニズム**:
  - アクセストークン有効期限: 1時間
  - リフレッシュトークン有効期限: 30日
  - バックグラウンドでの自動更新（アクセストークン期限の75%経過時）

- **セッション期限**:
  - アイドルタイムアウト: 30分
  - 最大セッション時間: 12時間

### 2.3 認可モデル

#### 2.3.1 ロールベースアクセス制御(RBAC)

**基本ロール**:
- **ゲストユーザー**: 未認証ユーザー、限定的な閲覧のみ可能
- **一般ユーザー**: 認証済みユーザー、基本機能にアクセス可能
- **管理者ユーザー**: 高度な設定やモニタリング機能にアクセス可能

**詳細なIAMロール設計**:
1. **親アカウントのロール**:
   - `AiDevUnauthenticatedRole`: 未認証ユーザー用
   - `AiDevAuthenticatedUserRole`: 認証済み一般ユーザー用
   - `AiDevAdministratorRole`: 管理者ユーザー用

2. **子アカウントのロール**:
   - `CustomerViewRole`: 顧客の閲覧用ロール
   - `CustomerManageRole`: 顧客の環境管理用ロール
   - `AgentExecutionRole`: AIエージェント実行用ロール

#### 2.3.2 アクセス制御ポリシー

- **リソースベースのポリシー**: S3やDynamoDBなどのサービス特有のアクセス制御
- **アイデンティティベースのポリシー**: ユーザーやロールに付与する権限
- **セッションポリシー**: 一時的に付与される制限付き権限

#### 2.3.3 多層アクセスモデル

aiDevシステムでは、以下の多層アクセスモデルを実装します：

1. **パブリックアクセス層**:
   - 認証不要でアクセス可能
   - 初期相談やヒアリング機能
   - マーケティングコンテンツ閲覧

2. **認証済みアクセス層**:
   - Cognitoによる認証必須
   - プリセールスエージェントとの対話
   - AWS環境設計の提案

3. **管理者アクセス層**:
   - 二要素認証(MFA)必須
   - 環境構築や高度な設定
   - 複数顧客の管理

4. **システム管理層**:
   - 特権アカウントによるアクセスのみ
   - システム全体の設定と監視
   - アカウント間の権限管理

## 3. マルチテナント設計

### 3.1 AWS Organizations構成設計

#### 3.1.1 組織構造

aiDevシステムでは、以下のAWS Organizations構造を採用します：

```
aiDev AWS Organization
├── 親アカウント（管理アカウント）
├── 共通サービスOU
│   ├── ログアーカイブアカウント
│   └── セキュリティ管理アカウント
└── 顧客OU
    ├── 顧客A-アカウント
    ├── 顧客B-アカウント
    └── 顧客N-アカウント
```

#### 3.1.2 サービスコントロールポリシー(SCP)

以下のSCPを定義して組織全体のセキュリティを確保します：

1. **ベースラインセキュリティSCP**:
   - すべてのアカウントに適用
   - セキュリティサービスの無効化を防止
   - 重要リージョン以外でのリソース作成を制限

2. **顧客アカウント制限SCP**:
   - 顧客OUに適用
   - 許可されたサービスのみ使用可能
   - 予算超過時のリソース作成制限

3. **データ保護SCP**:
   - 暗号化されていないデータの保存を禁止
   - 機密データの外部送信を制限

### 3.2 子アカウント自動プロビジョニング

#### 3.2.1 プロビジョニングワークフロー

ユーザーのサインアップ完了後、以下のワークフローで子アカウントを自動作成します：

1. Step Functionsワークフローの開始
2. Organizations APIを使用した子アカウント作成
3. 初期IAMリソースのセットアップ
4. CloudFormationスタックのデプロイ
5. ベースラインセキュリティ設定の適用
6. ユーザーとアカウントの関連付け

```state-machine
StartExecution
    |
    v
CheckExistingAccount
    |
    +----[アカウント存在]---> UpdateUserRecord
    |
    v[アカウント不存在]
CreateAccount
    |
    v
WaitForAccountCreation
    |
    v
SetupIAMResources
    |
    v
DeployBaseline
    |
    v
AssociateUserWithAccount
    |
    v
NotifyCompletion
```

#### 3.2.2 API設計

子アカウント自動プロビジョニングに使用する主要APIエンドポイント:

| エンドポイント | メソッド | 説明 |
|--------------|---------|------|
| `/api/accounts` | POST | 新規アカウント作成リクエスト |
| `/api/accounts/{id}` | GET | アカウント情報取得 |
| `/api/accounts/{id}/status` | GET | アカウント作成ステータス確認 |
| `/api/accounts/{id}/credentials` | GET | 一時的認証情報の取得 |

#### 3.2.3 顧客メタデータ管理

各子アカウントに関連付けられる顧客メタデータを親アカウントのDynamoDBで管理します：

**顧客アカウントテーブル**:
- **主キー**: `userId` (Cognito User ID)
- **ソートキー**: `accountId` (AWS アカウントID)
- **属性**:
  - `accountAlias` - アカウント表示名
  - `email` - 関連メールアドレス
  - `creationDate` - 作成日時
  - `status` - アカウント状態
  - `tier` - サービスティア
  - `settings` - 設定情報

### 3.3 クロスアカウント認証・認可

#### 3.3.1 アカウント間アクセスモデル

1. **ユーザーから子アカウントへのアクセス**:
   - Cognitoで認証後、STSを使用して子アカウントの一時的認証情報を取得
   - WebIdentity連携によるシームレスなアクセス

2. **親アカウントから子アカウントへのアクセス**:
   - サービスロールを使用した子アカウントリソースへのアクセス
   - 最小権限の原則に基づいた権限設定

#### 3.3.2 ロール信頼関係

子アカウントに以下のIAMロールを作成し、親アカウントとの信頼関係を確立します：

1. **クロスアカウント管理ロール**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::{ParentAccountId}:root"
      },
      "Action": "sts:AssumeRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalOrgID": "{OrganizationId}"
        }
      }
    }
  ]
}
```

2. **ユーザーアクセスロール**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "cognito-identity.amazonaws.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "cognito-identity.amazonaws.com:aud": "{IdentityPoolId}"
        },
        "ForAnyValue:StringLike": {
          "cognito-identity.amazonaws.com:amr": "authenticated"
        }
      }
    }
  ]
}
```

#### 3.3.3 一時的認証情報管理

1. **認証情報のライフサイクル**:
   - 有効期間: 1時間
   - 更新ロジック: 有効期限の75%経過時
   - バックグラウンド更新メカニズム

2. **セキュリティ対策**:
   - セッションポリシーによる権限制限
   - クライアント側での安全な保存（メモリのみ）
   - 認証情報の暗号化処理

### 3.4 マルチテナントデータ分離

#### 3.4.1 データ分離モデル

1. **物理的分離**:
   - 顧客データは各顧客の子アカウント内に保持
   - アカウント間のデータアクセスは明示的に許可された場合のみ

2. **論理的分離**:
   - 共通サービス内では顧客IDによるパーティショニング
   - DynamoDBではパーティションキーに顧客IDを使用

#### 3.4.2 共有リソースの分離

1. **S3バケット**:
   - バケットポリシーによるアクセス制限
   - プレフィクス/ディレクトリによる論理的分離

2. **DynamoDB**:
   - テナントIDをパーティションキーとして使用
   - 細粒度アクセス制御(FGAC)によるデータフィルタリング

3. **Bedrock API**:
   - テナント別のAPIキーと使用量管理
   - 会話履歴の分離管理

#### 3.4.3 テナント間通信の制御

1. **直接通信の禁止**:
   - テナント間の直接通信を防止
   - すべての通信は親アカウントを経由

2. **共有データの管理**:
   - 共有可能なデータの明示的定義
   - 共有プロセスの監査と記録

## 9. 実装詳細

### 9.1 Cognito認証フロー実装詳細

#### 9.1.1 Backend Lambda実装

サインアップ処理を実装したLambda関数の例:

```javascript
// AWS SDK importの例
const AWS = require('aws-sdk');
const cognito = new AWS.CognitoIdentityServiceProvider();
const stepFunctions = new AWS.StepFunctions();
const dynamoDB = new AWS.DynamoDB.DocumentClient();

// ユーザーサインアップ処理を行うLambda関数
exports.handler = async (event) => {
  try {
    const { email, password, userAttributes } = JSON.parse(event.body);
    
    // Cognitoユーザープールへの登録
    const signUpParams = {
      ClientId: process.env.COGNITO_CLIENT_ID,
      Username: email,
      Password: password,
      UserAttributes: [
        {
          Name: 'email',
          Value: email
        },
        // 必要に応じて追加属性
        ...Object.keys(userAttributes || {}).map(key => ({
          Name: key,
          Value: userAttributes[key]
        }))
      ]
    };
    
    const signUpResult = await cognito.signUp(signUpParams).promise();
    const userId = signUpResult.UserSub;
    
    // DynamoDBにユーザー情報を保存
    const userItem = {
      userId,
      email,
      creationDate: new Date().toISOString(),
      status: 'PENDING_CONFIRMATION',
      tier: 'FREE',
      settings: {}
    };
    
    await dynamoDB.put({
      TableName: process.env.USERS_TABLE,
      Item: userItem
    }).promise();
    
    // ユーザーが確認されたらAWSアカウント作成を開始するための
```

```javascript
import { Auth } from 'aws-amplify';

// トークンの自動更新を設定するユーティリティ
export const setupTokenRefresh = () => {
  let refreshTimeout = null;
  
  const scheduleRefresh = async () => {
    try {
      // 現在のセッション情報を取得
      const session = await Auth.currentSession();
      const idToken = session.getIdToken();
      const expirationTime = idToken.getExpiration() * 1000; // ミリ秒に変換
      const currentTime = Date.now();
      
      // 有効期限の75%経過時点でリフレッシュを行う
      const timeToRefresh = expirationTime - currentTime;
      const refreshTime = timeToRefresh * 0.75;
      
      // タイムアウトをクリア
      if (refreshTimeout) {
        clearTimeout(refreshTimeout);
      }
      
      // 新しいタイムアウトを設定
      refreshTimeout = setTimeout(async () => {
        try {
          await Auth.currentSession({ bypassCache: true }); // トークンを更新
          scheduleRefresh(); // 次の更新をスケジュール
        } catch (error) {
          console.error('Error refreshing token:', error);
          // エラー処理（再ログイン画面へのリダイレクトなど）
        }
      }, refreshTime);
    } catch (error) {
      console.error('Error setting up token refresh:', error);
    }
  };
  
  // 初回スケジュール設定
  scheduleRefresh();
  
  // クリーンアップ関数を返す
  return () => {
    if (refreshTimeout) {
      clearTimeout(refreshTimeout);
    }
  };
};
```

```javascript
const AWS = require('aws-sdk');
const sts = new AWS.STS();
const dynamoDB = new AWS.DynamoDB.DocumentClient();

exports.handler = async (event) => {
  try {
    const { userId, accountId } = JSON.parse(event.body);
    
    // ユーザーと子アカウントの関連を確認
    const userAccountResult = await dynamoDB.get({
      TableName: process.env.USER_ACCOUNTS_TABLE,
      Key: {
        userId,
        accountId
      }
    }).promise();
    
    if (!userAccountResult.Item) {
      return {
        statusCode: 403,
        body: JSON.stringify({ message: 'User does not have access to this account' })
      };
    }
    
    // STSを使用して子アカウントのロールを引き受ける
    const assumeRoleParams = {
      RoleArn: `arn:aws:iam::${accountId}:role/CustomerManageRole`,
      RoleSessionName: `${userId}-session`,
      DurationSeconds: 3600 // 1時間
    };
    
    const assumeRoleResult = await sts.assumeRole(assumeRoleParams).promise();
    const credentials = assumeRoleResult.Credentials;
    
    // 有効期限を計算
    const expirationTime = credentials.Expiration.getTime();
    const currentTime = Date.now();
    const timeToLive = Math.floor((expirationTime - currentTime) / 1000);
    
    // 認証情報をセッションに保存
    return {
      statusCode: 200,
      body: JSON.stringify({
        accessKeyId: credentials.AccessKeyId,
        secretAccessKey: credentials.SecretAccessKey,
        sessionToken: credentials.SessionToken,
        expiration: credentials.Expiration.toISOString(),
        timeToLive
      })
    };
  } catch (error) {
    console.error('Error assuming role:', error);
    return {
      statusCode: error.statusCode || 500,
      body: JSON.stringify({
        message: error.message || 'Error getting temporary credentials'
      })
    };
```

#### 9.3.2 クライアント側での一時的認証情報の利用

```javascript

import AWS from 'aws-sdk';
import { Auth } from 'aws-amplify';
import axios from 'axios';

// 子アカウントの一時的認証情報を取得
export const getAccountCredentials = async (accountId) => {
  try {
    // 認証トークンを取得
    const session = await Auth.currentSession();
    const idToken = session.getIdToken().getJwtToken();
    
    // APIを呼び出して子アカウントの認証情報を取得
    const response = await axios.post('/api/accounts/credentials', {
      accountId
    }, {
      headers: {
        Authorization: `Bearer ${idToken}`
      }
    });
    
    const credentials = response.data;
    
    // 認証情報をセッションストレージに保存（メモリのみ）
    sessionStorage.setItem(`account-credentials-${accountId}`, JSON.stringify({
      ...credentials,
      timestamp: Date.now()
    }));
    
    return credentials;
  } catch (error) {
    console.error('Error getting account credentials:', error);
    throw error;
  }
};

// AWSサービスクライアントを作成する関数
export const createAWSClient = async (accountId, service) => {
  try {
    // 保存された認証情報があるか確認
    const storedCredentialsJSON = sessionStorage.getItem(`account-credentials-${accountId}`);
    let credentials;
    
    if (storedCredentialsJSON) {
      const storedCredentials = JSON.parse(storedCredentialsJSON);
      const expirationTime = new Date(storedCredentials.expiration).getTime();
      const currentTime = Date.now();
      
      // 認証情報の有効期限が75%以上残っているか確認
      if (expirationTime - currentTime > (storedCredentials.timeToLive * 1000 * 0.25)) {
        credentials = storedCredentials;
      }
    }
    
    // 認証情報がない、または有効期限が近い場合は新しい認証情報を取得
    if (!credentials) {
      credentials = await getAccountCredentials(accountId);
    }
    
    // AWS SDKクライアントを設定
    const clientConfig = {
      region: 'ap-northeast-1', // または適切なリージョン
      credentials: {
        accessKeyId: credentials.accessKeyId,
        secretAccessKey: credentials.secretAccessKey,
        sessionToken: credentials.sessionToken
      }
    };
    
    // 指定されたサービスのクライアントを作成して返す
    return new AWS[service](clientConfig);
  } catch (error) {
    console.error(`Error creating AWS ${service} client:`, error);
    throw error;
  }
```

## 11. 本番環境に向けた考慮事項

### 11.1 スケーラビリティの確保

1. **データベースパーティショニング**:
   - 大規模テナントのデータをシャーディングして分散
   - 特定のテナントの負荷増大がシステム全体に影響しないよう分離

2. **認証サービスのキャパシティ管理**:
   - Cognitoの同時リクエスト制限に注意し、必要に応じてレート制限を実装
   - 複数リージョンでのユーザープール配置を検討

3. **子アカウント作成の並列処理**:
   - AWS Organizationsの制限（同時アカウント作成リクエスト制限）を考慮
   - 作成キューの実装とスロットリング

### 11.2 セキュリティの強化

1. **監査ログ強化**:
   - 全認証イベントのCloudWatch Logsへの記録
   - クロスアカウントアクセスのログ分析と異常検知

2. **顧客データの完全分離**:
   - 各顧客の重要データは専用の子アカウントに保管し、親アカウントには最小限のメタデータのみ保存
   - クロスアカウントアクセスで使用する権限を最小化

3. **セキュリティモニタリング**:
   - GuardDutyを全アカウントにデプロイし、集約監視
   - SecurityHubによる組織全体のセキュリティステータス管理

### 11.3 コスト最適化

1. **DynamoDB容量最適化**:
   - オンデマンドモードとプロビジョンドモードの使い分け
   - アクセスパターンに応じたインデックス設計

2. **Lambda関数の最適化**:
   - コールドスタート対策（Provisioned Concurrencyの検討）
   - メモリ割り当ての最適化による実行時間短縮

3. **S3ライフサイクルポリシー**:
   - 長期保存データの低頻度アクセス層への自動移行
   - 不要データの自動削除/アーカイブ

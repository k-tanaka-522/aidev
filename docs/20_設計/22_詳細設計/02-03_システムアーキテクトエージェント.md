# システムアーキテクトエージェント詳細設計

## 1. 概要

システムアーキテクトエージェントは、aiDevシステムにおいて技術的な設計と実装指針を担当するAIエージェントです。クラウドアーキテクチャ設計、非機能要件の具体化、セキュリティ設計、インフラストラクチャアズコード（IaC）の生成など、テクニカルアーキテクチャの専門家として、技術的に堅牢かつスケーラブルなシステム設計を提供します。

## 2. 役割と機能

### 2.1 主要な役割

1. **アーキテクチャ設計**
   - ビジネス要件に基づくクラウドアーキテクチャの設計
   - マイクロサービス構成の設計
   - スケーラビリティとパフォーマンスの設計

2. **非機能要件の具体化**
   - 可用性、信頼性、パフォーマンス要件の具体的な実現方法の設計
   - 運用性と保守性を考慮した設計
   - コスト最適化を考慮した設計判断

3. **セキュリティアーキテクチャ**
   - セキュリティ要件に基づく防御設計
   - アクセス制御と認証・認可の設計
   - データ保護とコンプライアンス対応

4. **IaC生成**
   - CloudFormationテンプレートの設計と生成
   - インフラ自動構築のためのコード化
   - 環境間の一貫性確保

5. **アプリケーション開発アーキテクチャ**
   - クロスプラットフォーム開発技術の選定と設計
   - フロントエンド/バックエンドアーキテクチャの設計
   - モバイル/デスクトップ/Web連携アーキテクチャの設計

6. **技術動向分析と適用**
   - 最新技術トレンドの継続的な評価と適用
   - プロジェクト要件に適したフレームワーク選定
   - 技術的負債の評価と最適なマイグレーション戦略の提案

7. **テクニカルガイダンス**
   - ベストプラクティスの提供
   - 技術選択の根拠説明
   - 設計上の制約と対応策の提示

## 4. プロンプト設計

### 4.1 システムプロンプト

```
あなたはシステムアーキテクトとして、クラウドシステムとアプリケーションの技術設計とアーキテクチャ策定を担当します。以下の役割と責務を持ちます：

1. ビジネス要件を技術要件に変換し、最適なアーキテクチャを設計する
2. AWSサービスの特性を理解し、適切なサービス選択と構成を提案する
3. AWS Well-Architectedフレームワークの5つの柱に基づいた堅牢な設計を行う
4. セキュリティ、可用性、パフォーマンス、コスト効率のバランスを最適化する
5. インフラストラクチャコードとして実装可能な詳細設計を提供する
6. 適切なアプリケーション開発技術とフレームワークを選定・設計する
7. 最新の技術トレンドを評価し、プロジェクトへの適用方法を提案する

設計プロセスでは、以下のポイントを心がけてください：
- ベストプラクティスと設計パターンを適切に適用する
- 設計判断とトレードオフを明確に説明する
- スケーラビリティと将来の拡張性を考慮する
- 運用の容易さとモニタリング可能性を重視する
- マネージドサービスを優先し、運用負荷を軽減する
- クロスプラットフォーム開発の可能性を常に検討する
- 技術選定では成熟度と将来性のバランスを考慮する

また、設計には以下の要素を必ず含めてください：
- リージョン/AZ戦略
- ネットワーク設計（VPC、サブネット、ルーティング）
- アクセス制御とIAM設計
- データストレージと永続化戦略
- バックアップとDR（災害復旧）計画
- モニタリングと運用設計
- アプリケーションアーキテクチャ（必要な場合）
- フロントエンド/バックエンド技術選定（必要な場合）
- デバイス間連携設計（必要な場合）
```

## 5. 状態管理

### 5.1 エージェント状態

システムアーキテクトエージェントは以下の状態を持ちます：

1. **要件理解状態**
   - 機能要件と非機能要件の把握
   - 制約条件の確認
   - 必要に応じた追加情報の要求

2. **アーキテクチャ設計状態**
   - 全体構成の策定
   - コンポーネントの選定
   - トレードオフの検討

3. **設計詳細化状態**
   - 各コンポーネントの詳細設計
   - インターフェース設計
   - データフロー設計

4. **IaC生成状態**
   - CloudFormationテンプレートの生成
   - 設定パラメータの設計
   - デプロイ手順の策定

5. **設計評価状態**
   - Well-Architectedフレームワークによる評価
   - リスク分析
   - 改善提案の作成

### 5.2 状態遷移図

```
[要件理解状態] --> [アーキテクチャ設計状態] --> [設計詳細化状態] --> [IaC生成状態] --> [設計評価状態]
     ^                     |                     |                                      |
     |                     |                     |                                      |
     +---------------------+---------------------+--------------------------------------+
```

※ 状態間は必要に応じて双方向に遷移可能

### 5.3 コンテキスト管理

- **要件コンテキスト**: 機能要件、非機能要件、制約条件
- **設計コンテキスト**: 設計判断、選定したAWSサービス、アーキテクチャパターン
- **評価コンテキスト**: レビュー結果、発見されたリスク、改善提案
- **IaCコンテキスト**: 生成したコード、パラメータ、デプロイ情報

## 7. エラーハンドリング

### 7.1 想定されるエラーパターン

1. **要件不明確エラー**
   - 設計に必要な要件情報が不足している場合
   - 対応: 特定の要件カテゴリに関する追加質問

2. **技術的矛盾エラー**
   - 要件間に技術的な矛盾がある場合
   - 対応: 矛盾点の明示とトレードオフの提示

3. **実現可能性エラー**
   - 要件が技術的に実現困難な場合
   - 対応: 制約の説明と代替アプローチの提案

4. **リソース制約エラー**
   - 予算や運用リソースと要件のギャップ
   - 対応: 段階的実装プランやサービスレベルの調整提案

### 7.2 エラー応答例

| エラーパターン | 応答例 |
|--------------|--------|
| 要件不明確エラー | 「アーキテクチャ設計を進めるために、{{特定要件}}についてもう少し詳しい情報が必要です。特に{{具体的な質問項目}}について教えていただけますか？」 |
| 技術的矛盾エラー | 「ご要望の{{要件A}}と{{要件B}}の間に技術的なトレードオフがあります。例えば、{{具体例}}のような状況が考えられます。優先すべき要件をお知らせいただけますか？」 |
| 実現可能性エラー | 「ご希望の{{要件}}は現在のAWSサービスでは以下の理由で完全な実現が困難です：{{理由}}。代替として{{代替案}}というアプローチがあり、これにより{{メリット}}を実現できます。」 |
| リソース制約エラー | 「この設計要件を満たすには、推定で月額{{コスト}}程度のAWSコストが発生します。予算制約内に収めるために、{{代替案}}というアプローチも検討できます。こちらでは{{機能制限}}という制約がありますが、コストを{{削減率}}削減できます。」 |

3. **CloudFormationテンプレート集**
   - 再利用可能なコンポーネントテンプレート
   - 環境タイプ別のテンプレート
   - ベストプラクティス実装例

   - サービス別の制限値
   - スケーラビリティの境界条件

5. **セキュリティベストプラクティス**
   - サービス別のセキュリティ設定

   - セキュリティアーキテクチャパターン

8. **技術トレンド情報**
   - 最新フレームワークの評価レポート

   - 技術採用リスク分析フレームワーク

## 9. パフォーマンス指標

### 9.1 品質指標

1. **アーキテクチャ品質**
   - Well-Architectedスコア
   - リスク検出数と重要度
   - 設計の完全性評価

2. **設計効率**
   - 要件カバー率
   - 再利用コンポーネント活用率
   - ベストプラクティス適用率

3. **実装可能性**
   - IaC変換精度
   - デプロイ自動化率
   - 運用負荷予測精度

### 9.2 効率指標

1. **設計速度**
   - 要件分析から設計完了までの時間
   - 主要決定ポイント数と決定速度

2. **適応性**
   - 要件変更への対応速度
   - 設計変更の影響範囲の最小化

3. **コミュニケーション効率**
   - 説明の明確さと理解度
   - 技術的な質問解決率

## 10. デプロイメント設計

### 10.1 実行環境

- AWS Lambda (Node.js 18.x)
- メモリ: 2048MB
- タイムアウト: 45秒
- 同時実行数: 最大30

### 10.2 スケーリング戦略

- 需要ベースの自動スケーリング
- 処理負荷に応じたメモリサイズの動的調整
- 大規模アーキテクチャ設計時の分散処理

### 10.3 監視設定

- CloudWatch Logsによる処理ログ収集
- 処理時間と品質スコアの追跡
- エラー率とリトライ頻度の監視

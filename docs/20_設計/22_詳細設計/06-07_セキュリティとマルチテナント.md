# aiDev セキュリティとマルチテナント性の両立に関する詳細設計

## 1. 概要

本ドキュメントは、aiDevシステムにおけるセキュリティとマルチテナント性の両立に関する詳細設計を定義します。aiDevでは、複数の顧客（テナント）が同一のシステム上でサービスを利用する環境において、高度なセキュリティと顧客データの厳格な分離を実現するための設計を提供します。

## 2. マルチテナントモデルの方針

### 2.1 マルチテナントアーキテクチャの選択

aiDevシステムでは、下記の考慮点を基に「Isolation-Focused Hybrid Model（分離重視ハイブリッドモデル）」を採用します：

| アーキテクチャモデル | 利点 | 欠点 | 採用判断 |
|------------------|-----|------|--------|
| 完全共有モデル<br>（1つのアプリケーション・DBで全テナント管理） | - コスト効率が高い<br>- 運用負荷が低い | - テナント間分離が弱い<br>- カスタマイズ性に制限 | ✗ |
| 完全分離モデル<br>（テナントごとに専用のインフラを提供） | - テナント間の完全分離<br>- 高度なカスタマイズ性 | - コスト効率が低い<br>- 運用負荷が高い | ✗ |
| **分離重視ハイブリッドモデル**<br>（アプリケーションは共有、データストアは分離） | - 強固なデータ分離<br>- コスト効率と安全性のバランス | - 一部運用負荷が発生<br>- アーキテクチャが複雑化 | ✓ |

### 2.2 分離重視ハイブリッドモデルの実装

#### 2.2.1 論理構造

```
+--------------------------------------+
|          共有コンポーネント           |
|  +-------------------------------+   |
|  |    フロントエンドアプリケーション   |   |
|  +-------------------------------+   |
|  +-------------------------------+   |
|  |         API Gateway          |   |
|  +-------------------------------+   |
|  +-------------------------------+   |
|  |      Lambda（制御レイヤー）     |   |
|  +-------------------------------+   |
+--------------------------------------+
                |
                | テナントID/コンテキストに基づくルーティング
                v
+--------------------------------------+
|          分離データレイヤー           |
| +----------------+ +---------------+ |
| | テナントA専用   | | テナントB専用  | |
| | AWSアカウント   | | AWSアカウント  | |
| |                | |               | |
| | - S3ストレージ  | | - S3ストレージ | |
| | - DynamoDB    | | - DynamoDB   | |
| | - Bedrock設定  | | - Bedrock設定 | |
| +----------------+ +---------------+ |
+--------------------------------------+
```

#### 2.2.2 AWS Organizationsを活用した実装

AWS Organizationsを使用し、以下の構成を実現します：

1. **親アカウント**：
   - 共有アプリケーションコンポーネント
   - 認証・認可サービス
   - テナント管理機能

2. **子アカウント（テナントごと）**：
   - 顧客ごとのデータストア
   - 顧客固有の設定
   - 顧客専用のサービス設定

## 3. データ分離メカニズム

### 3.1 階層的データ分離アプローチ

aiDevでは多層防御の原則に基づき、以下の3つのレベルでデータ分離を実装します：

1. **物理的分離（アカウントレベル）**
   - AWS Organizationsによる完全分離
   - アカウント間の明示的なアクセス制御

2. **論理的分離（IAMポリシーレベル）**
   - ロールベースのアクセス制御
   - きめ細かなIAMポリシー

3. **アプリケーション分離（コンテキストレベル）**
   - セッション内のテナントコンテキスト
   - テナントIDに基づくアクセスフィルタリング

### 3.2 データストア別の分離戦略

aiDevの主要データストアごとに、以下の分離戦略を実装します：

#### 3.2.1 S3バケット分離

**基本方針**: テナントごとに専用バケットを子アカウントに作成

```
# テナントごとのS3バケット構造
s3://tenant-{tenant_id}-data-{env}/
├── conversations/          # 会話履歴
├── generated-content/      # 生成コンテンツ
│   ├── architectures/      # アーキテクチャ図
│   ├── iac-templates/      # IaCテンプレート
│   └── documents/          # 生成ドキュメント
├── knowledge-base/         # テナント固有のナレッジベース
└── user-uploads/           # ユーザーアップロードファイル
```

**アクセス制御**:
- バケットポリシーによるクロスアカウントアクセス制限
- KMSによる暗号化（テナント固有のキー）
- タグベースのアクセス制御

#### 3.2.2 DynamoDB分離

**アプローチ1: 物理的テーブル分離（子アカウント内）**
- テナントごとに独立したDynamoDBテーブルを子アカウントに作成
- テーブル名にテナントIDを含める（例: `tenant-{tenant_id}-sessions`）
- 利点: 完全な分離、テナントごとのスケーリング
- 欠点: 運用複雑性の上昇

**アプローチ2: 論理的分離（親アカウント内）**
- 共有テーブルのパーティションキーにテナントIDを含める
- IAMポリシーによるアクセス制限（テナント別）
- 利点: 管理の簡素化、コスト効率
- 欠点: 分離の強度がIAMポリシーに依存

**採用方針**: アプローチ1（物理的分離）を基本とし、一部の共通データのみアプローチ2を適用

#### 3.2.3 Bedrockアクセスの分離

**分離方法**:
- 各子アカウントに専用のAPIキーを割り当て
- 会話履歴とプロンプトを子アカウント内に保持
- 使用量モニタリングとクォータをテナントごとに設定

## 4. クロスアカウントアクセス制御

### 4.1 IAMロール構造

親アカウントから子アカウントへのアクセスを制御するIAMロール構造：

```
親アカウント
├── TenantsManagementRole（全テナント管理用）
├── TenantAccessRole-A（テナントA専用アクセス用）
└── TenantAccessRole-B（テナントB専用アクセス用）

子アカウント（テナントA）
├── ParentAccountAccessRole（親アカウントからのアクセス用）
│   └── 信頼ポリシー: TenantAccessRole-Aのみ許可
└── AgentExecutionRole（AIエージェント実行用）

子アカウント（テナントB）
├── ParentAccountAccessRole（親アカウントからのアクセス用）
│   └── 信頼ポリシー: TenantAccessRole-Bのみ許可
└── AgentExecutionRole（AIエージェント実行用）
```

### 4.2 最小権限の原則適用

各ロールには、必要最小限の権限のみを付与します：

**TenantAccessRole-X（親アカウント）**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "sts:AssumeRole",
      "Resource": "arn:aws:iam::{TenantAccountId}:role/ParentAccountAccessRole",
      "Condition": {
        "StringEquals": {
          "aws:PrincipalTag/TenantId": "{TenantId}"
        }
      }
    }
  ]
}
```

**ParentAccountAccessRole（子アカウント）**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:ListBucket",
        "dynamodb:Query",
        "dynamodb:GetItem",
        "dynamodb:PutItem"
      ],
      "Resource": [
        "arn:aws:s3:::tenant-{TenantId}-*",
        "arn:aws:dynamodb:{Region}:{AccountId}:table/tenant-{TenantId}-*"
      ]
    }
  ]
}
```

### 4.3 セッション分離メカニズム

ユーザーセッションごとのテナント分離を確保する仕組み：

1. **セッションタグの使用**:
   - Cognito認証後、テナントIDをセッションタグとして付与
   - IAMポリシーで`aws:PrincipalTag/TenantId`条件を使用

2. **一時的認証情報の制御**:
   - STSの`AssumeRole`時に最小権限のセッションポリシーを適用
   - 有効期間の短いトークン（1時間以内）

3. **セッション追跡**:
   - 全セッションのアクティビティをCloudTrailで記録
   - テナントIDとユーザーIDによる関連付け

## 5. 暗号化戦略

### 5.1 暗号化階層

データの機密性を確保するため、以下の階層的暗号化戦略を実装します：

1. **保存データの暗号化**:
   - S3: SSE-KMS（テナント固有のCMK使用）
   - DynamoDB: テーブル全体の暗号化とフィールドレベルの暗号化の併用
   - パラメータストア: KMSによる暗号化

2. **転送中データの暗号化**:
   - 全通信でTLS 1.3を使用
   - APIゲートウェイでのHTTPSエンドポイントのみ許可
   - VPC内部通信の暗号化

3. **キー管理**:
   - テナントごとのマスターキー（CMK）管理
   - キーの自動ローテーション（年次）
   - マルチリージョンキー（災害対策用）

### 5.2 顧客マスターキー（CMK）管理

各テナントのデータ暗号化に使用するKMSキーの管理方法：

```
KMS キー階層
├── 組織マスターキー（親アカウント）
│   └── 用途: 管理メタデータの暗号化
├── テナントAマスターキー（テナントA子アカウント）
│   ├── 用途: テナントAのデータ暗号化
│   └── 管理: 親アカウントとテナントAのみがアクセス可能
└── テナントBマスターキー（テナントB子アカウント）
    ├── 用途: テナントBのデータ暗号化
    └── 管理: 親アカウントとテナントBのみがアクセス可能
```

キー管理ポリシー例（テナントAマスターキー）:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::{ParentAccountId}:role/TenantAccessRole-A"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey*"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::{TenantAccountId}:role/AgentExecutionRole"
      },
      "Action": [
        "kms:Encrypt",
        "kms:Decrypt",
        "kms:GenerateDataKey*"
      ],
      "Resource": "*"
    }
  ]
}
```

## 6. 監査とコンプライアンス

### 6.1 包括的なロギング戦略

組織全体のセキュリティ監査を可能にする包括的なロギング戦略：

1. **集約ログ収集**:
   - CloudTrail組織全体の証跡設定
   - 中央のログアーカイブアカウントへの転送
   - S3ライフサイクルポリシーによる長期保存

2. **ログ種別**:
   - APIアクティビティログ（CloudTrail）
   - リソースコンプライアンスログ（Config）
   - セキュリティファインディングログ（GuardDuty, SecurityHub）
   - アプリケーションログ（CloudWatch Logs）
   - データアクセスログ（S3アクセスログ、DynamoDBストリーム）

3. **ログフォーマット標準化**:
   - JSON構造化ログ
   - 共通フィールド定義（タイムスタンプ、テナントID、ユーザーID、アクション）
   - セッションIDによる関連付け

### 6.2 セキュリティモニタリングとアラート

セキュリティイベントの検出とアラート体制：

1. **リアルタイムモニタリング**:
   - GuardDutyによる脅威検出
   - 組織全体のSecurityHubによる集約
   - CloudWatch Eventsによるアラートトリガー

2. **アラート条件**:
   - クロスアカウント異常アクセス
   - KMSキー使用量の急増
   - 保護データへのアクセスパターン変化
   - 認証失敗の閾値超過

3. **対応プロセス自動化**:
   - Security Incident Response Workflow（Step Functions）
   - 重大度に応じた通知ルート（SNS, Email, SMS）
   - 自動遮断機能（重大インシデント時）

### 6.3 コンプライアンス対応

様々な規制要件に対応するための機能：

1. **コンプライアンスフレームワーク**:
   - GDPR（データ主体の権利対応）
   - HIPAA（医療情報のセキュリティ）
   - PCI DSS（支払情報保護）
   - ISO 27001/27017/27018

2. **コンプライアンス機能**:
   - データ保持ポリシー設定
   - 顧客データのエクスポート（データポータビリティ）
   - 顧客データの完全削除（忘れられる権利）
   - 監査証跡と変更証跡（監査対応）

## 8. リスクと緩和策

### 8.1 セキュリティリスク評価

| リスク | 影響度 | 発生可能性 | 緩和策 |
|------|------|---------|------|
| テナント間データ漏洩 | 高 | 低 | - 物理分離の徹底<br>- IAMポリシーの定期的レビュー |
| 認証基盤の脆弱性 | 高 | 低 | - MFAの必須化<br>- セッション有効期限の短縮 |
| 権限昇格 | 高 | 中 | - 定期的な権限監査<br>- 最小権限の徹底 |
| マスターキー漏洩 | 極高 | 極低 | - 厳格なキー管理<br>- キーの定期ローテーション |
| 管理者特権の悪用 | 高 | 低 | - 責務分離<br>- 特権アクセスの監査 |

### 8.2 運用リスクと対策

1. **複雑性に起因するリスク**:
   - リスク: マルチアカウント管理の複雑さによるヒューマンエラー
   - 対策: 自動化されたアカウント管理とポリシー適用

2. **スケーラビリティ課題**:
   - リスク: テナント数の増加に伴う管理負荷の増大
   - 対策: Infrastructure as Code（IaC）によるプロビジョニング

3. **コスト予測困難**:
   - リスク: 子アカウント数増加によるコスト増大
   - 対策: 詳細なコスト分析と予算アラートの設定

### 8.3 緊急対応計画

セキュリティインシデント発生時の対応計画：

1. **検知フェーズ**:
   - 異常検知メカニズム（GuardDuty, SecurityHub）
   - アラート発報と初期評価

2. **封じ込めフェーズ**:
   - 影響アカウント/テナントの分離
   - 一時的なアクセス制限
   - データアクセスの遮断

3. **撲滅・復旧フェーズ**:
   - 侵害リソースの特定と削除
   - 安全なバックアップからの復元
   - アクセス権限の再設定

4. **教訓化・報告フェーズ**:
   - インシデント分析とレポート作成
   - システム強化策の実施
   - 必要に応じたテナント通知

## 9. 実装詳細とベストプラクティス

### 9.1 AWS Organizations SCP設計

テナント環境のセキュリティを強化するサービスコントロールポリシー（SCP）：

**ベースラインセキュリティSCP**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": [
        "organizations:LeaveOrganization",
        "iam:DeleteRolePermissionsBoundary",
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail"
      ],
      "Resource": "*"
    },
    {
      "Effect": "Deny",
      "Action": [
        "kms:DeleteKey",
        "kms:ScheduleKeyDeletion",
        "kms:DisableKey"
      ],
      "Resource": "arn:aws:kms:*:*:key/*",
      "Condition": {
        "StringEquals": {
          "aws:ResourceTag/Purpose": "CustomerData"
        }
      }
    }
  ]
}
```

**データ保護SCP**:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": [
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    },
    {
      "Effect": "Deny",
      "Action": [
        "dynamodb:CreateTable"
      ],
      "Resource": "*",
      "Condition": {
        "Null": {
          "dynamodb:TableClass": "true"
        }
      }
    }
  ]
}
```

## 10. まとめと次のステップ

aiDevシステムのセキュリティとマルチテナント性の両立に関する詳細設計を定義しました。この設計に基づき、フェーズ別の実装計画に従って、安全かつ分離されたマルチテナント環境を構築していきます。

次のステップとして、以下の作業を進めます：

1. **初期アカウント構造のセットアップ**:
   - 組織の作成と親アカウントの構成
   - ベースラインセキュリティの適用

2. **自動プロビジョニングメカニズムの実装**:
   - アカウント作成自動化
   - 標準ポリシーの適用

3. **クロスアカウントアクセス制御の実装**:
   - IAMロール設定
   - 最小権限ポリシーの適用

これらの実装と並行して、セキュリティ監視体制の整備とコンプライアンス対応を進め、安全で堅牢なマルチテナントシステムを実現します。

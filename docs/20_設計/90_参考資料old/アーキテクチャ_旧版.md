# aiDev アーキテクチャ設計書

## 1. アーキテクチャ概要

aiDevシステムは、AWS Bedrockを中核技術として活用したAI開発支援窓口（プリセールスエージェント）です。本設計書では、システムの全体アーキテクチャ、主要コンポーネント、データフロー、およびセキュリティ設計について定義します。

### 1.1 アーキテクチャの目標

- AWS Bedrockを最大限に活用した対話型AIエージェントの実現
- サーバーレスアーキテクチャによるスケーラビリティと低運用コストの両立
- セキュアで高可用性を持つシステム基盤の構築
- 段階的な機能拡張が容易なモジュラー設計
- 継続的インテグレーション・デプロイによる迅速なイテレーション

### 1.2 全体アーキテクチャ

aiDevシステムは以下の主要コンポーネントで構成されます：

```
[ユーザー] → [CloudFront] → [S3静的Webサイト] ← → [API Gateway] → [Lambda] → [AWS Bedrock]
                                                      ↑            ↓
                 [Cognito] ← → [認証/認可] ← →        └── [DynamoDB/S3]
                                                           [その他AWS Services]

[開発者] → [CodeCommit] → [CodePipeline] → [CodeBuild] → [デプロイ先環境]
                ↑             ↓                ↓
                └── [開発環境] [テスト環境]     [本番環境]
```

#### 主要技術コンポーネント

1. **フロントエンド**
   - S3静的ウェブホスティング（OSSのChatbot UI）
   - CloudFront（コンテンツ配信と証明書管理）
   - WAF（Webアプリケーションファイアウォール）

2. **認証・認可**
   - Amazon Cognito（ユーザー管理、認証）
   - IAM（アクセス制御）

3. **バックエンド（APIレイヤー）**
   - API Gateway（RESTful API）
   - Lambda（ビジネスロジック、Bedrock連携）

4. **AI処理**
   - AWS Bedrock（大規模言語モデル）
   - Bedrock Knowledge Base（ナレッジベース）

5. **データストア**
   - S3（対話履歴、ナレッジデータ保存）
   - DynamoDB（メタデータ、ユーザー設定等）

6. **環境構築自動化**
   - CloudFormation（IaC）
   - Lambda（ワークフロー自動化）

7. **監視・運用**
   - CloudWatch（ログ、メトリクス、アラーム）
   - X-Ray（分散トレーシング）
   - CloudTrail（API監査）

8. **CI/CD**
   - CodeCommit（ソースコード管理）
   - CodePipeline（パイプライン管理）
   - CodeBuild（ビルドプロセス）
   - ECR（コンテナイメージリポジトリ）

## 2. コンポーネント詳細設計

### 2.1 フロントエンドコンポーネント

#### 2.1.1 技術構成
- **静的Webホスティング**: Amazon S3
- **コンテンツ配信**: Amazon CloudFront
- **Webアプリケーション**: OSSのChatbot UIをベースにカスタマイズ
- **フレームワーク**: React/Next.js

#### 2.1.2 主要機能
- 対話インターフェース（チャットUI）
- 認証フロー（Cognitoとの連携）
- 対話履歴表示
- ファイルアップロード（ナレッジデータ）
- レスポンシブデザイン（PC/モバイル対応）

#### 2.1.3 設計上の考慮点
- SPA（Single Page Application）として実装
- ローカルストレージを活用した一時データキャッシュ
- API Gateway経由でバックエンドと通信
- CORS設定による安全な通信

### 2.2 認証・認可コンポーネント

#### 2.2.1 技術構成
- **ユーザー管理・認証**: Amazon Cognito
- **アクセス制御**: IAM
- **認証フロー**: OAuth 2.0/OpenID Connect

#### 2.2.2 主要機能
- ユーザー登録・ログイン
- ソーシャルIDプロバイダー連携（オプション）
- 多要素認証
- 権限管理（一般ユーザー/管理者）

#### 2.2.3 設計上の考慮点
- JWTトークンを用いたAPIアクセス認証
- トークンの適切な有効期限設定
- Cognitoトリガーを活用したカスタム認証フロー
- IAMポリシーによる細かなアクセス制御

### 2.3 バックエンドコンポーネント（APIレイヤー）

#### 2.3.1 技術構成
- **API管理**: Amazon API Gateway
- **ビジネスロジック**: AWS Lambda
- **通信プロトコル**: REST API
- **データ形式**: JSON

#### 2.3.2 主要API
- **対話API**: Bedrockとの対話を処理
- **履歴API**: 対話履歴の取得/保存
- **設定API**: ユーザー設定管理
- **環境構築API**: 環境自動構築の要求/状態確認

#### 2.3.3 設計上の考慮点
- API GatewayとLambda統合によるサーバーレスAPI実装
- Lambda関数の適切な分割（単一責任の原則）
- コールドスタート問題への対応（Provisioned Concurrency等）
- API Gateway ステージを活用した環境分離（開発/テスト/本番）

## 3. データフロー

### 3.1 ユーザー対話フロー

1. ユーザーがフロントエンドでCognito認証を行う
2. フロントエンドがAPI Gatewayを通じてLambdaを呼び出す
3. LambdaがBedrockにリクエストを送信
4. Bedrockが必要に応じてナレッジベースを検索
5. Bedrockからの応答をLambdaが処理
6. 処理結果がAPI Gatewayを通じてフロントエンドに返却
7. 対話ログがS3に保存される

### 3.2 環境構築フロー

1. ユーザーが環境構築リクエストを発行
2. LambdaがBedrockを用いて要件分析
3. 分析結果に基づいてCloudFormationテンプレートを生成
4. 生成したテンプレートをS3に保存
5. CloudFormationを用いて環境構築を実行
6. 構築状態をDynamoDBで管理
7. 完了通知をユーザーに送信

### 3.3 CI/CDフロー

1. 開発者がコードをCodeCommitリポジトリにプッシュ
2. CodePipelineがソース変更を検出し、パイプラインを起動
3. CodeBuildがソースコードのビルドとテストを実行
   - アプリケーションビルド: フロントエンド/バックエンドコードのビルド
   - コンテナビルド: Dockerイメージの構築とECRへのプッシュ
   - インフラビルド: CloudFormationテンプレートの検証とパッケージング
4. ビルド成功後、開発環境へのデプロイを自動実行
5. テスト実行と検証
6. 手動承認後、ステージング/本番環境へデプロイ
7. デプロイ後の監視とアラート

## 8. 技術的制約と対策

### 8.1 AWS Bedrock関連の制約と対策

- コンテキスト長の制限: コンテキスト圧縮・要約機能の実装
- APIコール制限: 適切なスロットリングとキューイング
- モデル制限: 使用目的に最適なモデル選択
- 応答時間: 非同期処理とプログレッシブ表示の実装

### 8.2 サーバーレスアーキテクチャの制約と対策

- Lambda実行時間制限: 長時間処理の分割と状態管理
- コールドスタート: Provisioned Concurrency活用
- ステートレス設計: 外部ストアでの状態管理
- 開発・テスト環境: ローカルエミュレーション環境の整備

### 8.3 CI/CD関連の制約と対策

- ビルド時間制限: 効率的なビルドプロセスの設計
- 並行ビルド制限: 重要なパイプラインの優先度設定
- アーティファクトサイズ制限: 適切な分割とパッケージング
- 環境間の差異: 環境設定の外部化とパラメータ化

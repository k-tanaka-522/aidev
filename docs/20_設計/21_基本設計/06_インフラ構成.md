# aiDev インフラ構成

## 1. インフラ構成の概要

aiDevシステムは、AWS上に構築されるサーバーレスアーキテクチャを採用し、スケーラビリティとコスト効率を両立させた構成です。本ドキュメントでは、システムを支えるインフラストラクチャのコンポーネント、接続、セキュリティ設計について記述します。

## 2. 全体アーキテクチャ

aiDevシステムは以下のAWSサービスを主要コンポーネントとして採用します：

- **フロントエンド**: S3, CloudFront
- **API**: API Gateway, Lambda
- **認証**: Cognito
- **データベース**: DynamoDB
- **ストレージ**: S3
- **AIエンジン**: AWS Bedrock
- **CI/CD**: CodeCommit, CodePipeline, CodeBuild
- **監視**: CloudWatch, X-Ray

## 3. コンポーネント詳細

### 3.1 フロントエンドインフラ

**主要サービス**:
- **Amazon S3**: 静的Webサイトホスティング
- **Amazon CloudFront**: CDNによる高速配信
- **AWS WAF**: Webアプリケーションファイアウォール

**構成詳細**:
- S3バケット設定:
  - 静的ウェブサイトホスティング有効化
  - 直接アクセス禁止（CloudFrontのみ許可）
  - バージョニング有効化

- CloudFront設定:
  - カスタムドメイン名とSSL証明書
  - キャッシュポリシー（HTTPヘッダーに基づく）
  - エッジロケーション最適化

- WAF設定:
  - SQLインジェクション保護
  - XSS攻撃保護
  - レートベース制限（DDoS対策）

### 3.2 APIインフラ

**主要サービス**:
- **API Gateway**: RESTful API、WebSocket API
- **Lambda**: サーバーレス関数
- **IAM**: 認証と認可

**構成詳細**:
- API Gateway設定:
  - リージョン型API
  - カスタムドメイン
  - スロットリング設定
  - APIキー管理
  - ステージ設定（開発/テスト/本番）

- Lambda設定:
  - 関数サイズとメモリ割り当て
  - 実行タイムアウト
  - 言語ランタイム: Node.js 18.x
  - 同時実行数制限
  - 環境変数（環境別設定）

### 3.3 認証・認可インフラ

**主要サービス**:
- **Amazon Cognito**: ユーザープール、IDプール
- **IAM**: ロールとポリシー

**構成詳細**:
- Cognito設定:
  - ユーザープール設定（サインアップ/サインイン）
  - 多要素認証(MFA)
  - ソーシャルIDプロバイダー連携（オプション）
  - カスタム認証フロー

- IAM設定:
  - 最小権限の原則に基づくロール定義
  - リソースベースのポリシー
  - セッションポリシー

### 3.4 データストアインフラ

**主要サービス**:
- **DynamoDB**: NoSQLデータベース
- **S3**: オブジェクトストレージ
- **Bedrock Knowledge Base**: ナレッジベース

**構成詳細**:
- DynamoDB設定:
  - オンデマンドキャパシティモード
  - テーブル構成（2.4.1参照）
  - DAXクラスター（オプション、キャッシュ用）
  - バックアップ設定

- S3設定:
  - バケットポリシー
  - ライフサイクルルール
  - アクセスポイント
  - 暗号化設定

- ナレッジベース設定:
  - ベクトルデータベース
  - インデックス更新頻度
  - クエリパラメータ

### 3.5 AI処理インフラ

**主要サービス**:
- **AWS Bedrock**: LLMサービス
- **Lambda**: AIオーケストレーション

**構成詳細**:
- Bedrock設定:
  - モデル選択（Claude 3.7/3.5, Hermes等）
  - 推論パラメータ

- Lambda AIオーケストレーション:
  - プロンプト管理
  - コンテキスト処理
  - 応答フォーマット

### 3.6 CI/CDインフラ

**主要サービス**:
- **CodeCommit**: ソースコードリポジトリ
- **CodePipeline**: パイプライン管理
- **CodeBuild**: ビルドプロセス
- **ECR**: コンテナリポジトリ

**構成詳細**:

  - ソース、ビルド、デプロイステージ
  - 承認ステップ
  - 環境分離（開発/テスト/本番）

- CodeBuild設定:
  - ビルド環境
  - IAM権限
  - アーティファクト管理

### 3.7 監視・運用インフラ

**主要サービス**:
- **CloudWatch**: モニタリング、ログ管理
- **X-Ray**: 分散トレーシング
- **CloudTrail**: API監査
- **EventBridge**: イベント処理

**構成詳細**:
- CloudWatch設定:
  - カスタムメトリクス
  - アラーム設定
  - ロググループと保持期間
  - ダッシュボード

- X-Ray設定:
  - サンプリングルール
  - アノテーション
  - グループとフィルタ

- CloudTrail設定:
  - 監査対象イベント
  - ログ保存と検索

## 4. ネットワーク設計

### 4.1 VPCデザイン

フェーズ1ではサーバーレス構成を中心とし、VPC内リソースを最小限に抑えます。フェーズ2以降で以下の要素を含むVPC設計を導入：

- **サブネット構成**:
  - パブリックサブネット（2AZ以上）
  - プライベートサブネット（2AZ以上）
  - データベースサブネット（2AZ以上）

- **セキュリティグループ**:
  - API層セキュリティグループ
  - アプリケーション層セキュリティグループ
  - データ層セキュリティグループ

- **ネットワークACL**:
  - パブリックサブネット用NACL
  - プライベートサブネット用NACL

### 4.2 エンドポイント設計

- **VPCエンドポイント**:
  - S3ゲートウェイエンドポイント
  - DynamoDBゲートウェイエンドポイント
  - Bedrockインターフェースエンドポイント
  - その他AWS必要サービスのエンドポイント

- **API公開設計**:
  - API Gateway：インターネット経由でアクセス可能
  - プライベートAPI（フェーズ2以降）

## 5. セキュリティ設計

### 5.1 暗号化戦略

- **保存データ暗号化**:
  - S3: SSE-S3（AES-256）
  - DynamoDB: AWS KMSによる暗号化
  - Lambda環境変数: KMSによる暗号化

- **転送中データ暗号化**:
  - TLS 1.3使用
  - 最新の暗号スイート適用
  - HTTPS強制

- **シークレット管理**:
  - AWS Secrets Manager活用
  - 定期的なローテーション

### 5.2 アクセス制御

- **認証メカニズム**:
  - Cognitoベースの認証
  - JWTトークン検証
  - API Keyと使用量制限（外部サービス用）

- **認可モデル**:
  - IAMロールによる権限制御
  - リソースベースのポリシー
  - 属性ベースのアクセス制御（ABAC）

- **最小権限の原則**:
  - 各Lambdaに必要最小限の権限
  - 細分化されたIAMポリシー

### 5.3 防御メカニズム

- **DDoS対策**:
  - CloudFront配信
  - WAFレートベース制限
  - Shield Standardの活用

- **脆弱性対策**:
  - セキュリティパッチの自動適用
  - 依存関係スキャン
  - コード静的解析

- **侵入検知**:
  - CloudTrailによる監査
  - GuardDutyによる異常検知（オプション）
  - 不正アクセスアラート

## 8. AWS Organizations連携設計

### 8.1 基本アーキテクチャ

aiDevシステムでは、AWS Organizationsを活用した以下のアーキテクチャを採用します：

- **親アカウント(管理アカウント)**: システム全体の管理、ユーザー認証、子アカウント作成を担当
- **子アカウント**: 顧客ごとに作成され、プリセールスエージェントと環境構築を実行
- **アカウント分離**: 顧客データは各子アカウント内に閉じて管理し、機密性を確保

### 8.2 アカウント構造

```
aiDev AWS Organization
├── 親アカウント (管理アカウント)
│   ├── 共通インフラストラクチャ
│   ├── アカウント管理サービス
│   ├── ユーザー認証サービス
│   └── モニタリング・管理サービス
│
├── サービス管理OU
│   ├── ログアーカイブアカウント
│   └── セキュリティ管理アカウント
│
└── 顧客用OU
    ├── 顧客Aアカウント
    │   ├── プリセールスエージェント
    │   ├── 顧客データ
    │   └── 構築環境
    ├── 顧客Bアカウント
    │   └── ...
    └── 顧客Nアカウント
        └── ...
```

### 8.3 簡易アカウント作成フロー

**ユーザー操作フロー**:
1. メールアドレスのみを入力して登録開始
2. 確認メールを受信し、検証リンクをクリック
3. 初期パスワード設定と最小限のプロファイル情報入力
4. プリセールスエージェントへの接続完了

**システム内部プロセス**:
1. メールアドレス登録時に一時的なユーザーレコードを作成
2. メール検証後に永続的なユーザーアカウント作成を開始
3. AWS Organizations APIを使用して子アカウント作成プロセスを開始
4. 子アカウント作成完了後にクロスアカウントロールを設定
5. プリセールスエージェントを子アカウントにデプロイ
6. ユーザーに子アカウントへのアクセス権を付与

### 8.4 子アカウント自動プロビジョニング

**アカウント作成ワークフロー**:
1. AWS Organizations APIを使用した子アカウント作成要求
2. 基本設定の適用（アカウント名、連絡先Eメール等）
3. 初期リソース制限の設定
4. クロスアカウントロールの作成
5. セキュリティベースラインの適用
6. プリセールスエージェント用インフラの自動デプロイ

**使用するAWSサービス**:
- AWS Organizations
- AWS IAM
- AWS CloudFormation
- AWS Lambda
- AWS Step Functions

### 8.5 親子アカウント間の連携

**認証連携**:
- 親アカウントのCognitoで認証
- 子アカウントアクセスのための一時的クレデンシャル発行
- 子アカウント内のロールへのフェデレーション

**クロスアカウント権限モデル**:
- 環境管理ロール: 子アカウントの環境設定を管理
- デプロイメントロール: プリセールスエージェントのデプロイに使用
- モニタリングロール: 使用状況と健全性の監視に使用

**データ連携**:
- 共通ナレッジベースへの参照
- 使用統計と課金情報の収集
- イベント駆動型の非同期データ同期

### 8.6 セキュリティとコンプライアンス

**アカウント分離によるセキュリティ向上**:
- 顧客データの厳格な分離
- アカウントレベルでの権限境界の実現
- 認証情報の分離管理

**通信セキュリティ**:
- すべての通信はTLS 1.3で暗号化
- API呼び出しの署名検証
- JWTトークンによる認証と有効期限管理

**データセキュリティ**:
- 保存データの暗号化（S3, DynamoDB）
- KMSによる暗号化キー管理
- 機密データの子アカウント内封じ込め

## 9. フェーズ別実装計画

### 9.1 フェーズ1（MVP）

- 基本的なサーバーレスアーキテクチャ
- CloudFront + S3 + API Gateway + Lambda + DynamoDB
- メールアドレスのみの簡易登録と子アカウント自動作成の基本機能
- シンプルな認証（Cognito）
- 基本的な監視（CloudWatch）

### 9.2 フェーズ2

- VPC導入とプライベート連携の強化
- 完全な自動プロビジョニングプロセス
- 拡張されたセキュリティ設定の自動適用
- WAF強化
- 高度な監視と自動対応
- コスト最適化機能の追加

### 9.3 フェーズ3

- マルチリージョン対応（必要に応じて）
- 高度なDisaster Recovery機能
- コンテナベースのワークロード（必要に応じて）
- AI処理の最適化と分散
- 複数企業・組織の管理機能
- 包括的な監査とレポート機能

# aiDev システム連携

## 1. システム連携の概要

aiDevシステムでは、複数のコンポーネントとサービスが連携して動作します。本ドキュメントでは、これらのコンポーネント間の連携方式、インターフェース、およびデータフローについて定義します。

## 2. 主要コンポーネント間連携

### 2.1 フロントエンド ⟷ バックエンド連携

**連携方式**:
- REST API（API Gateway経由）
- WebSocket（リアルタイム更新用）

**主要インターフェース**:
- ユーザー認証・認可
- セッション管理
- メッセージ送受信
- 成果物取得・更新

**データフォーマット**:
- JSON形式のリクエスト/レスポンス
- バイナリデータ（ファイルアップロード/ダウンロード）用のマルチパートフォーム

**セキュリティ要件**:
- JWT認証トークン
- HTTPS通信
- CORS設定

### 2.2 バックエンド ⟷ AIエンジン連携

**連携方式**:
- AWS Bedrock API
- Lambda関数間連携

**主要インターフェース**:
- AIモデル呼び出し
- コンテキスト情報の受け渡し
- 応答生成と処理

**データフォーマット**:
- AIモデル専用のプロンプト形式
- JSON形式のコンテキストデータ
- バイナリ形式の添付ファイル参照

**最適化要件**:
- プロンプトキャッシング
- 応答ストリーミング
- コンテキスト圧縮

### 2.3 AIエンジン ⟷ ナレッジベース連携

**連携方式**:
- Bedrock Knowledge Base API
- S3経由のコンテンツ参照

**主要インターフェース**:
- 関連情報検索
- コンテンツ取得
- 検索結果のプロンプト統合

**データフォーマット**:
- ベクトル検索クエリ
- 構造化メタデータ
- テキスト/画像/ドキュメントコンテンツ

**パフォーマンス要件**:
- インデックス最適化
- キャッシング戦略
- 検索結果ランキング

### 2.4 バックエンド ⟷ データストア連携

**連携方式**:
- AWS SDK経由のAPI呼び出し
- DynamoDB直接アクセス

**主要インターフェース**:
- CRUD操作
- クエリとスキャン
- トランザクション処理

**データフォーマット**:
- DynamoDB項目形式
- S3オブジェクト
- バイナリラージオブジェクト

**最適化要件**:
- 読み取り/書き込みキャパシティ最適化
- インデックス戦略
- コスト効率の高いデータアクセスパターン

### 2.5 バックエンド ⟷ AWS環境構築連携

**連携方式**:
- CloudFormation API
- AWS SDK経由のリソース操作

**主要インターフェース**:
- テンプレート生成と検証
- スタック作成・更新・削除
- リソース状態監視

**データフォーマット**:
- YAML/JSON形式のCloudFormationテンプレート
- リソース構成パラメータ
- 状態変化イベント

**セキュリティ要件**:
- 最小権限の原則
- パラメータ暗号化
- 変更審査プロセス

## 4. データフロー

### 4.1 ユーザー対話フロー

```
ユーザー → フロントエンド → API Gateway → Lambda → Bedrock → Lambda → DynamoDB/S3 → API Gateway → フロントエンド → ユーザー
```

**フロー詳細**:
1. ユーザーがメッセージを入力
2. フロントエンドがAPIを呼び出し
3. Lambdaがメッセージをコンテキストと共にBedrockに送信
4. Bedrockが必要に応じてナレッジベースを検索
5. Bedrockの応答をLambdaが処理
6. 応答と対話履歴をデータストアに保存
7. 処理結果をフロントエンドに返却
8. フロントエンドが応答を表示

### 4.2 環境構築フロー

```
ユーザー → フロントエンド → API Gateway → Lambda → Bedrock → Lambda → S3 → Lambda → CloudFormation → Lambda → DynamoDB → API Gateway → フロントエンド → ユーザー
```

**フロー詳細**:
1. ユーザーが環境構築を要求
2. フロントエンドがAPIを呼び出し
3. Lambdaが要件をBedrockに送信
4. Bedrockが適切な設計を生成
5. LambdaがCloudFormationテンプレートを生成
6. テンプレートをS3に保存
7. LambdaがCloudFormationスタックを作成
8. 構築プロセスの状態をDynamoDBで追跡
9. 進捗と結果をフロントエンドに返却
10. フロントエンドが状態を表示

### 4.3 エージェント切替フロー

```
エージェントA → 内部API → エージェントオーケストレーター → 内部API → エージェントB
```

**フロー詳細**:
1. エージェントAが他のエージェントへの切替を判断
2. 切替要求とコンテキストをオーケストレーターに送信
3. オーケストレーターが適切なエージェントを選択
4. コンテキスト情報を新エージェントに転送
5. エージェントBが処理を引き継ぎ

## 5. API設計基本方針

### 5.1 REST API設計

**エンドポイント階層構造**:
- `/api/v1/users/` - ユーザー関連API
- `/api/v1/projects/` - プロジェクト関連API
- `/api/v1/sessions/` - セッション関連API
- `/api/v1/artifacts/` - 成果物関連API
- `/api/v1/environments/` - 環境関連API

**HTTP動詞の利用**:
- GET: リソース取得
- POST: リソース作成
- PUT: リソース更新
- DELETE: リソース削除
- PATCH: リソース部分更新

**標準レスポンス形式**:
```json
{
  "status": "success" | "error",
  "data": { /* レスポンスデータ */ },
  "error": { "code": "ERROR_CODE", "message": "エラーメッセージ" },
  "meta": { "page": 1, "total": 100, ... }
}
```

### 5.2 WebSocket API設計

**イベントタイプ**:
- `message` - チャットメッセージ
- `notification` - システム通知
- `status` - 状態更新
- `typing` - 入力状態

**メッセージ形式**:
```json
{
  "type": "イベントタイプ",
  "data": { /* イベントデータ */ },
  "timestamp": "ISO日時文字列"
}
```

## 6. エラーハンドリング

### 6.1 エラー分類

- **クライアントエラー** (4xx): ユーザー入力誤り、認証エラー等
- **サーバーエラー** (5xx): システム内部エラー、外部サービス障害等
- **ビジネスロジックエラー**: システム正常動作中の論理的エラー

### 6.2 エラー伝搬ポリシー

- フロントエンドでの適切なエラー表示
- 開発環境での詳細エラー情報提供
- 本番環境でのエラー詳細の適切な隠蔽
- 構造化ログによるエラー追跡性確保

### 6.3 再試行とフォールバック

- 一時的エラーの自動再試行
- AI応答エラー時の代替応答生成
- 段階的なフォールバック戦略
